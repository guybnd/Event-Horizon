## Event Horizon 1.1 — artifacts you can argue with, a redesigned cockpit, rock-solid sessions

Event Horizon is the development environment for the **agent side** of your work: break work into tickets, hand each to an agent that carries full context and runs on its **own git branch, in parallel**, and review everything as a **real pull request** before it lands. Tickets, plans, and history are plain markdown in your repo — local-first, no cloud, no account. It orchestrates an agent CLI you already use (Claude Code, Gemini CLI, GitHub Copilot CLI) or a supported IDE; it ships no LLM of its own.

**1.1 is the first feature release on the 1.0 line.** It adds rich, annotatable grooming **artifacts**, a redesigned **notification & pending cockpit**, and a deep reliability pass on the orchestrator's **realtime + human-in-the-loop** layers — plus a wave of upgrades to the MCP tools your agents actually call.

### Grooming artifacts — reason against a rendering, not prose

- Agents can publish a **self-contained HTML artifact** to a ticket — a mockup, an architecture/flow diagram (Mermaid), an SVG wireframe, a clickable Tailwind prototype, or visual acceptance criteria — rendered in a **sandboxed, opaque-origin iframe** (no network, cookies, or storage). You react to the real thing before code is written (#173).
- **Annotate any region.** Select text *or* right-click any element — toggles, chart bars, buttons that have no selectable text. Collect several notes, send them as one message, and they round-trip to the agent, which revises and republishes. A **revision picker** keeps the full history (#175, #185).
- A **layout-audit gate** masks the artifact until it passes conservative checks (overflow, off-canvas, clipped text, overlapping blocks), so artifacts stay visually clean (#178).
- Artifacts render **inline in chat the moment they're published** — no reload, no minimize/reopen dance — with a **"publishing…" indicator** during the large tool-input stream (#188, #192, #195, FLUX-924).

### A redesigned notification & pending cockpit

- The **notification panel was redesigned** from the ground up and unified with the require-input/pending surface into one loud "needs you" cockpit, with motion and polish (#196, #189).
- The **Activity log moved into a header panel** next to the pending surface (#174).
- **Parked agent sessions are now visually distinct** from active ones, so you can tell at a glance which sessions are waiting on you (#190, #191).

### Human-in-the-loop: inline, durable, never lost

- When a chat is already open, the agent's question now **draws inline in the conversation** instead of only in a separate window (#197).
- Require-Input / awaiting-input prompts **render full markdown** and are **scrollable + resizable**, with click-to-expand and click-outside-to-collapse (#183, #184, #187).
- Prompts **survive an engine restart** (per-process binding secret), and answers submitted from the board now **appear in the chat transcript** (#186, #165).
- The pending window dismisses itself when no prompts remain (FLUX-891).

### Realtime & session reliability

- The orchestrator chat stream **no longer silently stalls** — messages, "working" state, prompts, and stop all ride a **keepalived SSE stream** (#193).
- **Stop sticks**: stopped/resumed sessions no longer wedge "active" forever or revert to waiting-input (#193).
- Fixed the optimistic-bubble vanish, the dead "working" window, and queued-message double-dispatch in the chat client (#193).
- A dispatched session **no longer floods the orchestrator** with per-narration WORKING rows (#193).
- Hardening: dev-watcher hot-restart tree-kill, workspace-rebind desync, delegate double-fire, clear-transcript-on-error (#193).
- **Chat-window geometry persists** across minimize/reopen and reload (#194).

### Board activity & dispatch visibility

- A durable, filterable **Board Activity / History view** of dispatch events (#170).
- **DispatchChip polish**: lifecycle color-coding, a live "working" pulse, duration, timestamps, and hierarchy (#167).
- Dispatch-activity rows enriched with **phase, title, and a prominent timestamp** (#166).

### Sharper agent experience (the MCP tools your agents call)

- MCP tool results emit **compact JSON** (no pretty-print indentation) — cheaper and faster for agents (#171).
- **Contextual next-step hints** in tool responses, **definitive empty states** for list-style tools, and **machine-readable error codes** on failures (#180, #179, #172, #177).
- Oversized ticket bodies are **truncated in the agent view** with a size hint + a `fullBody` escape hatch (#181).
- `list_tickets` is **active-by-default** with a limit and search; `get_board_config` strips Tailwind color classes from the agent-facing payload (#199, #198).
- `nextStepForStatus` is now case-insensitive and config-driven (#182).

### Engine & infrastructure

- A **stderr-only structured logger** + a `console.log` sweep make MCP stdout safe by construction (#200).
- Engine-managed shared MCP servers are **keyed per worktree** (one Serena per worktree), ending cross-worktree thrash (#202).
- `copilot`/`gemini` `stop()` now **tree-kill**, at parity with `claude-code` (#164).
- A **cheaper model option** for delegated subagents (#201).
- Shared agent-adapter helpers extracted to `engine/src/agents/shared.ts`, plus a blocking-HITL routing fix (#204).
- Public-repo tree cleanup for production release (#168).

### Install

- **Desktop app** — download the `.dmg` (macOS, Apple Silicon) or `.exe` (Windows) below: Event Horizon in its own window with a taskbar/dock entry.
- **Tray binary** — a single executable that runs in the system tray; no Node required.
- **Run from source** — `event-horizon-source.zip` below; Node 20+.
- **Bring your own agent** — Claude Code, Gemini CLI, or GitHub Copilot CLI, or a supported IDE (Cursor, Windsurf, Cline).

> **First launch:** the builds are unsigned, so macOS Gatekeeper and Windows Defender may show a first-launch false positive. **macOS:** `xattr -d com.apple.quarantine <path>`, or right-click → Open. **Windows:** allow the file in Defender, or run from source. Full details in the README's first-run troubleshooting.
