## Release v0.10.1

### Highlights

MCP executable mode, ticket auto-repair on load, "Copy Fix Instructions" for parse errors, and a wave of agent-reliability improvements: subtask API, schema enforcement, and history normalization.

### Features

**MCP Executable Mode**
- `.mcp.json` now targets the compiled Windows executable with an `--mcp` flag instead of `npx tsx`
- Engine detects `--mcp` flag at startup and redirects stdout to stderr before any module loads, preventing JSON-RPC frame corruption
- `mcp-server.ts` auto-start guard: only fires when the file is the direct entry point, not when imported as a module
- MCP and HTTP server modes are fully separated â€” the same compiled binary handles both

**Ticket Auto-Repair on Load**
- `task-store.ts` now attempts to repair common schema violations before failing validation:
  - `oldStatus`/`newStatus` â†’ `from`/`to` rename for `status_change` history entries
  - Missing or unrecognized `type` inferred from entry shape (from/to â†’ status_change, comment field â†’ comment, sessionId â†’ agent_session)
  - Malformed or missing `date` values repaired or replaced with current timestamp
  - Missing `user` field defaulted to `"Unknown"`
  - Inline subtask objects with `id` fields extracted to string IDs
  - Missing `title` derived from filename
- Repaired tickets are written back to disk and a repair log is added to history
- Only falls back to a parse error if auto-repair is insufficient

**Parse Error â€” Copy Fix Instructions**
- Parse error modal now includes a "Copy Fix Instructions" button per broken ticket
- Copies a structured prompt with the file path, error description, expected YAML schema, and step-by-step fix instructions â€” ready to paste into an agent

**Subtask Creation API** (FLUX-278)
- New `POST /api/tasks/:parentId/subtasks` endpoint creates a child ticket and links it to the parent atomically
- Accepts `{ title, status?, priority?, effort?, body? }`, auto-assigns the next sequential ID
- `patch-ticket` CLI gains `--add-subtask <parentId>` flag for agent use

**Engine Schema Enforcement** (FLUX-287, FLUX-288, FLUX-289)
- New `engine/src/schema.ts` module: `validateTicketFrontmatter`, `validateHistoryEntry`, `validateSubtasks`
- Validation runs on every API write path (POST/PUT); malformed writes return 400 with details
- Validation runs on every read; invalid tickets surface in `parseErrors` (visible in portal) rather than crashing
- `normalizeHistoryEntries` now migrates legacy `oldStatus`/`newStatus` â†’ `from`/`to` and marks the ticket for rewrite â€” self-healing without a migration script
- Skill files updated with explicit WRONG vs RIGHT examples for `status_change` shape and timestamp format

### Bug Fixes

**Agent Ticket Persistence** (FLUX-274)
- Identified and fixed four root causes preventing agents from persisting ticket changes after session completion:
  - Skill files now include explicit instructions to write ticket changes through the engine API
  - Engine no longer flushes session progress to the ticket file during active agent sessions (prevented agent writes)
  - `buildInitialPrompt()` now handles `Grooming` status (previously fell through to a no-op generic message)
  - `updateAgentSession()` and `updateTaskWithHistory()` now re-read the full file from disk before writing, preventing stale-cache clobbers of agent edits

**Inline Subtask Auto-Creation** (FLUX-277)
- When an agent writes subtasks as inline YAML objects instead of string IDs, the engine now auto-creates the corresponding `.flux/<id>.md` files and rewrites the parent's `subtasks` array to string IDs
- Prevents portal crashes caused by unexpected subtask shapes; system is now self-healing

### Improvements

- `init.ts` copies `.docs/skills/` and `.flux/skills/` into new workspaces on init, making projects self-contained for the agent workflow without needing the EH source root
