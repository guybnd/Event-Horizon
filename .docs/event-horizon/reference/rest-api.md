# REST API Reference

Authoritative list of HTTP endpoints exposed by the engine on `http://localhost:3067`. The REST surface is primarily for the portal; agents should use the [MCP tools](mcp-tools.md) instead.

> Both REST and MCP read and write the same in-memory `tasksCache` in [`task-store.ts`](../../../engine/src/task-store.ts) and persist through the same atomic write path. Mutations broadcast SSE events on `/api/events` so the portal updates live.

## Conventions

- All routes are mounted under `/api/*` from [`engine/src/index.ts`](../../../engine/src/index.ts).
- Routes marked **workspace-scoped** below sit behind the `requireWorkspace` middleware. If no workspace is active, they return `503` with `{ error: 'NO_WORKSPACE' }`. The portal intercepts this and shows the workspace picker.
- Request bodies are JSON unless noted. Body size limit is 10 MB.
- Success responses are JSON unless noted.
- Mutation endpoints append entries to ticket history and broadcast SSE events (`taskCreated`, `taskUpdated`, `taskDeleted`).

## Server-level

| Method | Path | Workspace? | Purpose |
|--------|------|-----------|---------|
| GET | `/api/health` | no | `{ status: 'ok', workspace: <root \| null> }` |
| POST | `/api/shutdown` | no | Stop all CLI sessions and exit the process |
| GET | `/api/update-check` | no | Cached update-availability snapshot |

## Tasks (`/api/tasks`) — workspace-scoped

Sourced from [`engine/src/routes/tasks.ts`](../../../engine/src/routes/tasks.ts) and [`cli-session.ts`](../../../engine/src/routes/cli-session.ts).

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/tasks` | All tickets (array), or pass `?active=true` to get only non-terminal tickets (excludes `Done`/`Released`/the configured archive status — same definition as MCP `list_tickets`' active screen, `getTerminalStatuses()` in `task-store.ts`). **FLUX-970**: the portal's routine 3s poll uses `?active=true` — on a large board most tickets are resting Released/Archived/Done ones that never change, so shipping/parsing/diffing them every 3s was pure waste; the board already filters them out client-side anyway. Callers that need everything (Releases screen, global search) still use the unfiltered default, refreshed on mount/focus/SSE-triggered reconciles rather than every tick. **Conditional GET (FLUX-1144):** the response carries an `ETag` keyed on a monotonic version bumped by every task mutation (`taskCreated`/`taskUpdated`/`taskDeleted`, hooked into `broadcastEvent` in `events.ts`) **and** the `active` flag, since the two query variants serialize different sets. Send the ETag back as `If-None-Match`; an unchanged poll gets a bodyless **304** instead of a re-serialized list — the portal's `fetchTasks` does this automatically per query variant and treats a 304 as "keep current state". Each ticket carries a **capped** `cliSession`/`cliSessions[]`: the single most-recent session **and** the plural list (active sessions + the most-recent completed run group) both truncate `liveOutput` to a short tail (~2KB) — use `GET /api/tasks/:id/cli-session` or the detail endpoint for the full buffer. **History is replaced by a compact `historyDigest` (FLUX-725)** so a large board doesn't parse the full array on every poll/SSE. The digest pre-computes everything the cards + attention surfaces derive from history: `{ length, lastEntry:{date,type}, lastActivityAt, enteredCurrentStatusAt, isSpeedDemon, statusChanges24h:[{from,to,date}], comments:[{id,user,date}] (no text, always the FULL set — the source unread badges + "mark all read" read from), requireInput:{question,setDate}\|null, planReviewComment:{text,date,user}\|null (FLUX-1289/FLUX-1303 — the latest feedback comment, only populated while `planReviewState` is set; `user` attributes it: reviewer vs a human's own re-groom notes) }`. The list `history[]` is pruned to only the entry types a card renders **inline** — `comment` (the hover comment popover shows text + threaded replies) and the **active** `agent_session` (the card's inline live-progress, keyed by sessionId); every `status_change`, `activity`, and **terminal** `agent_session` (with its long `finalMessage`/`summary`/token metadata) is dropped. **FLUX-1144:** the inline `comment` entries are further capped to the most recent `commentDigest.keepRecent` (config, default 3) — a heavily-commented ticket was the single largest field in a full-board response; `historyDigest.comments` still carries every comment's id so unread counts and "mark all read" are never affected by the cap, only which comments render inline on hover (the popover flags when it's showing fewer than the total). Use `GET /api/tasks/:id` for the **full**, uncapped `history`. |
| GET | `/api/tasks/errors` | Parse errors keyed by file path — for the ParseError banner. |
| GET | `/api/tasks/:id` | Single ticket. Returns the **full** `cliSessions[]` (all sessions, full `liveOutput`). Append `?view=agent` (optional `&historyLimit=N`, `&expand=id1,id2`, `&fullHistory=true`, `&fullBody=true`) for the digested agent surface instead — same shape as MCP `get_ticket`: agent_session entries without `progress[]`, windowed history, oversized `body` truncated with a recoverable size hint (FLUX-879; `&fullBody=true` overrides), slimmed session summaries. Agents using REST as the MCP fallback should always pass `?view=agent`. |
| POST | `/api/tasks` | Create a ticket. Body: `{ author, title, status?, priority?, effort?, tags?, body?, assignee?, history?, projectKey?, ... }`. Allocates next id from `projectKey` (or first configured project), checks remote ids in orphan mode to avoid collisions, validates schema, writes atomically. Returns the created ticket. |
| POST | `/api/tasks/:parentId/subtasks` | Create a child ticket and link it from the parent. Body mirrors POST `/api/tasks` plus parent linkage. |
| PUT | `/api/tasks/:id` | Update a ticket. Body: any subset of metadata fields, optional `body`, optional `status`, optional `appendHistory: HistoryEntry[]` to append comments / activity entries. Used by the portal and as the REST fallback for agents when MCP is unavailable. Optional `skipCommentRequirement: boolean` (FLUX-847, never persisted — stripped from the update before write): when true, relaxes **only** the config-gated Ready comment check (`requireCommentOnStatusChange`); the Require Input comment check is a hard invariant and always rejects a comment-less move regardless of this flag. The portal sends it when the user's session "don't ask again" skip is on, so Ready transfers go through silently while Require Input still prompts. **History reconciliation (FLUX-1308):** prefer `appendHistory` — it's a pure delta, immune to a stale client snapshot. A caller MAY instead submit a full rebuilt `history` array (some legacy writers still do); the engine reconciles it against the server's copy by entry **identity** (`id` where present — comment/activity entries always have one; otherwise a content signature), not array length/position. Any entry the server already has is kept even if the submitted array omits it (never deleted by omission); any entry in the submitted array that doesn't match an existing one is treated as novel and appended, server-stamped with the current timestamp — regardless of how stale the client's base snapshot was or where in its array the new entries appear. (Before this fix, reconciliation was by length — `submitted.slice(existingLength)` — so a snapshot stale by N entries silently dropped its first N novel entries.) **FLUX-1044:** the write routes through the same per-ticket serialized atomic write path (`updateTaskWithHistory`, FLUX-645 lock) the MCP tools use, and the shared transition rules (comment gates, schema validation + tag registration sequencing) live in `status-transition-service.ts` — one rulebook for REST and MCP. The commit-before-Ready refusal stays deliberately MCP-only (FLUX-730/731): a portal drag-to-Ready is a visible human action and is not blocked. |
| DELETE | `/api/tasks/:id` | Delete a ticket. Removes the markdown file, unlinks from parent `subtasks`, deletes asset directory if present. **Tears down the ticket's dedicated worktree first** (FLUX-577) — stops its sessions + detaches the worktree (abandon path: uncommitted work preserved as a stash ref, not applied to master) so the delete doesn't orphan it. A shared/joined worktree another ticket holds is left alone; best-effort (teardown failure doesn't block the delete). |
| POST | `/api/tasks/:id/assets` | Upload an image asset. Body: `{ filename, contentBase64 }`. Writes under `.flux/assets/<ticket-id>/<unique-name>`. Returns the relative markdown path the caller should embed. **FLUX-676 / FLUX-1209:** `:id` also accepts either virtual conversation sentinel (`__board__` or `__furnace__`) — pasted images for the orchestrator or Furnace-chat land under `.flux/assets/__board__/` / `.flux/assets/__furnace__/` respectively (neither has a ticket sidecar); every other `:id` must be a real ticket. |
| POST | `/api/tasks/:id/branch` | Create a git branch for the ticket. Body: `{ baseBranch?, worktree? }`. When `worktree` is `true` (or omitted with config `worktreeByDefault` on) a dedicated git worktree is created for the branch (FLUX-516); response is `{ branch, worktree?, worktreeError? }`. |
| GET | `/api/tasks/:id/branch` | Get branch status (`{ name, exists, aheadCount, behindCount, worktree }`). `worktree` is the dedicated worktree's absolute path, or null (drives the portal Detach control). |
| DELETE | `/api/tasks/:id/branch` | Delete the ticket's branch. Body: `{ force? }`. Stops any live session and detaches the ticket's worktree first (a branch can't be deleted while a worktree holds it). As an **abandon**, uncommitted work is preserved as a stash ref but NOT applied onto master. |
| POST | `/api/tasks/:id/worktree/detach` | Remove the ticket's worktree but keep the branch (manual-finish escape hatch). Uncommitted work is preserved: stashed → applied onto master, or kept as a stash ref on conflict. Response `{ outcome: 'clean'\|'applied'\|'stashed', stashRef?, message }`. 404 when the ticket has no worktree. |
| POST | `/api/tasks/:id/worktree/open` | Ensure a branch + worktree exist and open the worktree in a **new VS Code window** (FLUX-522). Reuses an existing worktree already checked out on the branch (joined tickets) rather than creating a second. Response `{ worktree, branch, opened, seedPrompt }`; `opened` is false when the `code` CLI isn't on PATH (open `worktree` manually). |
| POST | `/api/tasks/:id/worktree/join` | Adopt an existing branch so this ticket runs in **that branch's worktree** (shared-branch work — e.g. fixing review-found bugs alongside the parent ticket). Body `{ branch }`. Sets the ticket's `branch` (no new branch/worktree); `409` when no worktree holds that branch. Response `{ branch, worktree, joined }`. |
| GET | `/api/tasks/:id/pr` | Live PR state for the ticket's **branch** (PRs are branch-scoped) via `gh pr view`. Response `{ pr: { number, state, url, title, reviewDecision, mergeable, checks: { total, passed, failed, pending } } }`, or `{ pr: null }` when the ticket has no branch, no PR exists, or `gh` is unavailable/unauthed (`getPullRequestStatus` returns null on any gh failure — no separate auth pre-check, FLUX-561). Best-effort — never 500. Powers the `PR-<n>` deck card / "Open PRs" group (FLUX-565; the FLUX-558 `open-pr` swimlane on normal tickets is retired — FLUX-569). |
| POST | `/api/tasks/:id/pr` | **Raise PR**: push the ticket's branch + open a PR for review **without** moving to Done (Done happens at merge). Reuses `branch-manager.createPullRequest`; stores the PR URL as `implementationLink`. Response `{ url, number }`. `409` when the ticket has no branch, or `{ error, unavailable: true }` when `gh` is unauthed. |
| POST | `/api/tasks/:id/retry` | **Retry a merged/closed PR** (FLUX-593). `:id` is a `kind:'pr'` ticket. Body `{ reason (required), createBranch?, updatedBy? }`. A merged PR is immutable, so this spawns a **new ticket** (status In Progress) linked to the PR via a `links: [{type:'retries', target:'PR-<n>'}]` relation, with a body carrying the reason + the PR's context (url/diff + original member ids) as agent launch-focus; optionally creates a `flux/<new-id>-retry-pr-<n>` branch. Response `{ id, branch }` (branch null if not requested/failed). `409` if `:id` isn't a PR ticket; `400` if no reason. |
| POST | `/api/tasks/:id/pr/update-branch` | Refresh a stale PR branch by merging the default branch into it (FLUX-559). Conservative: requires an **active worktree** holding the branch and a **clean tree** (`409` otherwise), and **aborts the merge on conflict** (`409` — the user resolves in the worktree) so the tree is never left half-merged. On success pushes the merge so the PR updates. Response `{ updated: true, branch }`. |
| POST | `/api/tasks/:id/pr/merge` | Squash-merge the branch's PR (`gh pr merge --squash`), then run post-merge cleanup (FLUX-557): advance **every** branch-sharing ticket → Done, fast-forward local master, and tear down the worktree + force-delete the branch when the tree is clean. **PR tickets (`kind:'pr'`) on the branch are resolved to Done + `prState:'MERGED'` immediately** (cleanup itself skips them — their state is owned by `syncPrTickets` — so without this the merged PR card would sit OPEN until the next 90s poll; FLUX-588). Response `{ merged: true, outcome: 'cleaned'\|'unsafe', branch, advanced: string[], masterSynced, worktreeRemoved, branchDeleted, reason?, notificationId? }`. A **dirty worktree** is kept (`outcome: 'unsafe'`) and a persistent notification with *Clean up / Open worktree* actions is raised — never silently discarded. **Two-tier session guard (FLUX-636):** Tier 1 — hard block: `409 { error }` when any branch-sharing ticket has a `pending` or `running` session (actively executing; merging would clobber in-flight work) — stop the session first. Tier 2 — parked sessions: `409 { error, parkedOnly: true, parkedOwners: string[] }` when the only blocking sessions are `waiting-input` (parked — work committed, nothing in-flight). Pass body `{ stopParkedSessions: true }` to auto-end those parked sessions and proceed with the merge; each parked owner gets an activity log entry. Ending a parked session loses only the warm `--resume`; committed work and cold-resume-from-transcript are unaffected. **Shared-PR guard (FLUX-569):** `409 { error, sharedNonDone: [{id,status,title}], requiresForce: true }` when the branch is shared by non-terminal sibling tickets a merge would sweep to Done — pass body `{ force: true }` to confirm and merge the whole shared PR anyway (the deck card lists the swept tickets in its merge-confirm, then re-sends with `force`). `409` when no branch / `gh` unavailable; `500` on merge failure. |
| POST | `/api/tasks/:id/finish` | **Engine-side finish for BRANCHLESS tickets (FLUX-618)** — the zero-token sibling of `/pr/merge` (branch/PR tickets finish by merging their PR; a branchless ticket needs a curated commit that used to require a tokenized agent `finish`). Body `{ message (required commit message), files: string[] (required — the EXPLICIT paths to stage), completionComment? (defaults to `message`) }`. Stages **only** the listed files (pathspec-scoped `git add -- <files>` then `git commit -m <message> -- <files>`, mirroring `/commit` — **never** `git add -A`), reads back the short hash, then runs the same finish tail as `finish_ticket`'s branchless path: completion comment, `implementationLink` = commit hash, swimlane cleared, lazy baseline repair (anchors a missing `baselineCommit` at `HEAD~1`) + diff capture + `.diff` sidecar, status → Done, reap stale parked sessions. Response `{ finished: true, hash, link }`. `404` when the ticket doesn't exist; `409` when the ticket **has a branch** (finish it via `/pr/merge`) or isn't in the `readyForMergeStatus` (`Ready`); `400` for a missing message, an empty file list, or an unsafe path (absolute / `..` / drive-letter); `500` on commit failure. |
| POST | `/api/tasks/:id/plan-review/start` | **Re-review plan** (FLUX-1289): run ONE plan-review pass on a `Grooming` ticket — the REST twin of the [`start_plan_review`](mcp-tools.md#start_plan_review) MCP tool, for a human clicking the portal button (`PlanApprovalPanel`; FLUX-1303 removed it from the dock/chat cards and gates the panel button on the plan having **changed** since the last verdict, via [`planReviewBodyHash`](ticket-schema.md)). Runs `startPlanGateNow(mode:'one-pass')` (never loops, any gate value); records its verdict to `planReviewState` without moving the ticket. No body. Response `{ ok: true, message }`; `409 { error, reason }` when not in Grooming / a pass is already in flight / Furnace-owned; `404` unknown ticket. |
| POST | `/api/tasks/:id/plan-review/revise` | **Send for re-grooming** (FLUX-1303): the atomic revise entry point behind every portal plan-review surface (AttentionDock item, chat card, `PlanApprovalPanel`). Body `{ notes?, user? }`. ONE call: records `notes` as a `user`-attributed comment (server-side append — immune to the stale-snapshot full-history replace that lost send-back notes, ex-FLUX-1301), stamps `planReviewState: 'changes-requested'` + [`planReviewBodyHash`](ticket-schema.md), clears `needsAction`, dispatches the grooming revise session seeded with the reviewer feedback pointer AND the notes, and registers the run with the plan-gate runner (`startPlanReviseNow`, `gate-runner.ts`) in the reimplementing state — so the revision is automatically re-reviewed when the session completes (`auto`→`loop-auto`, `auto-then-you`→`loop-confirm`; `you`→`one-pass`: exactly one re-review, then it stops and flags the human either way). Replaces the portal's old two-step revise (dispatch + follow-up PUT verdict clear) that could strand a stale changes-requested card (ex-FLUX-1302). Server-enforced guards: overriding an `approved` verdict REQUIRES `notes` (`400 reason:'notes-required'` — a bare REST call can't silently flip an approval with no stated reason), and a failed persist ABORTS the whole revise with nothing dispatched (`500 reason:'persist-failed'`) rather than running a session against a ticket whose notes/verdict never landed. Also clears a stale gate-park `require-input` swimlane (the revise IS the human's answer). Response `{ ok: true, message }`; `409 { error, reason }` when not in Grooming / a gate run is already in flight / Furnace-owned; `400 reason:'notes-required'` / `500 reason:'persist-failed'` for the two guards above; `404` unknown ticket. |
| POST | `/api/tasks/:id/pr/adopt` | **Continue development on a PR** by binding work to its branch (FLUX-569). `:id` is a `kind:'pr'` ticket (typically a zero-member / unattached PR opened directly on GitHub). Body `{ mode, updatedBy? }`: `mode:'adopt'` + `ticketId` rebinds an existing ticket to the PR's branch and moves it to **In Progress**; `mode:'create'` + `title` (+ optional `body`) creates a fresh ticket bound to the branch (status In Progress, linked via `links:[{type:'continues', target:'PR-<n>'}]`). Either way the new member is work-gated In Progress on the branch, so it folds into the deck immediately (members recomputed + stamped on the PR ticket). Response `{ memberId, members }`. `409` if `:id` isn't a PR ticket / has no branch, if the adopt target is itself a PR ticket, or if the adopt target is **already bound to a different branch** (adopting it would orphan it from its existing PR and abandon committed work — detach or create instead); `400` for an unknown mode or missing title; `404` if the adopt target doesn't exist. |
| GET | `/api/tasks/worktrees` | List active task worktrees: `{ worktrees: [{ path, branch, ticketId, ticketTitle, changedFiles, aheadCount }] }`. Each is mapped to the ticket whose branch it holds (null = orphan). `changedFiles` counts files differing from master (committed + uncommitted + untracked) — powers the board worktree chip's badge. `aheadCount` (FLUX-582) = commits this worktree is ahead of the default branch (`git -C <path> rev-list --count <default>..HEAD`, best-effort `0`); pairs with `changedFiles` for the Worktrees panel's `↑N · M vs master` divergence badge. Drives the worktree badges, board count/filter chip, Join + Attach pickers, and management panel. |
| GET | `/api/tasks/uncommitted-count` | Uncommitted state of the active workspace: `{ count, branch, diverged }`. `count` = working tree vs `HEAD` (tracked changes) + untracked, via `worktreeChangeCount(workspaceRoot, 'HEAD')`; `branch` = current branch name (`'HEAD'` if detached, `null` on error). `diverged` (FLUX-582) = sum over all task worktrees of `worktreeChangeCount(w.path, 'master')` (committed + uncommitted vs master), best-effort `0` — drives the stoplight's secondary `↑N vs master` segment (additive; tone + commit dropdown still keyed off `count`). Best-effort `0`/`null` when not a git repo. Powers the board-header uncommitted stoplight + dropdown (FLUX-535/544). |
| POST | `/api/tasks/open-editor` | Open VS Code (FLUX-544). Body `{ file?, ref? }`: with `file` (repo-relative, traversal-guarded) reveals that file via `code -g` in the correct checkout — `ref` is the diff group's ref (`'main'`/omitted → workspace root; a branch → that branch's worktree). Without `file`, opens the workspace root in a new window. Returns `{ opened }`; false when the `code` CLI isn't on PATH. |
| POST | `/api/tasks/commit` | Commit selected uncommitted files from the board panel (FLUX-554). **Commit-only — never pushes.** Body `{ ref, files: string[], message }`: `ref` picks the checkout (`'main'`/omitted → workspace root; a branch → that worktree); paths are repo-relative + traversal-guarded. Runs `git add -- <files>` then `git commit -m <message> -- <files>` (pathspec-scoped, so only the listed files commit). Returns `{ hash }` (new short SHA) or `{ error }` (git stderr) with 4xx/500. |
| POST | `/api/tasks/discard` | Discard selected files' **uncommitted** changes (FLUX-1333) — the destructive sibling of `/commit`, behind the portal's per-file "Discard change" controls (Changes screen, chat diff panel, uncommitted stoplight). Body `{ ref, files: string[] }` with the same `ref` addressing, but **tighter**: an unresolvable worktree refuses with `404` instead of falling back to the main tree (never discard in a guessed checkout). Refuses with `409` while an agent session is actively executing (`pending`/`running`) in the target tree. Per-state semantics via `git status --porcelain -uall` + hardened `git restore` (see [`workspace-discard.ts`](../../../engine/src/workspace-discard.ts)): modified/deleted → restore index+worktree from HEAD (clears mixed staged+unstaged); untracked → delete from disk; staged add → unstage + delete; staged rename → **atomic both sides** (old path restored, new path removed). Committed-only work is never touched — such a file yields a per-file error. Returns `{ results: [{ file, ok, error? }] }` (one failed file never aborts the rest); `400`/`404`/`409` for request-level refusals. |
| GET | `/api/tasks/branches` | List local branches for the "Attach to branch" picker: `{ branches: [{ name, hasWorktree, isTicketBranch }] }`. `hasWorktree` marks branches already checked out in a worktree; `isTicketBranch` marks branches owned by a ticket. |
| GET | `/api/tasks/:id/diff` | Fetch the unified diff for the ticket. Default (`?mode=committed`, or omitted) serves the sidecar captured at `finish_ticket`; 404 when no diff is stored. `?mode=working` generates a live diff on the fly against the working tree (`baselineCommit` vs working tree for branch-less tickets), 404 when no anchor is resolvable. Optional `?file=<path>` query returns only that file's hunk. Response is `text/plain`. |
| GET | `/api/tasks/:id/artifact` | Serve a revision of the ticket's rich artifact (FLUX-873 — a grooming mockup/diagram, or a FLUX-976 Ready-time visual recap), published via the `publish_artifact` MCP tool. `?rev=<n>` selects a revision; omitted/`?rev=latest` serves the newest (falls back to the highest file on disk if the pointer drifts). Response is the agent-authored **`text/html`** document, hardened for sandboxed-iframe rendering: strict `Content-Security-Policy` (`default-src 'none'`; inline + named CDNs allowed for script/style; `connect-src 'none'`, `form-action 'none'`, `base-uri 'none'`, `frame-ancestors 'self'`), `X-Content-Type-Options: nosniff`, `Cache-Control: no-store`, and `X-Artifact-Revision: <n>`. The HTML lives in a traversal-guarded sidecar at `.flux/artifacts/{ID}/{rev}.html`; `rev` must be a positive integer or `latest` (else `400`), and `404` when the ticket/revision has no stored artifact. The portal embeds this in `<iframe sandbox="allow-scripts">` (no `allow-same-origin`). **FLUX-874:** a fixed anchor-capture/annotation script (`injectAnnotatorScript`) is injected just before `</body>` at serve time (the stored file stays pristine; permitted by the existing `script-src 'unsafe-inline'`) — it powers the in-iframe region selection + the host↔iframe `postMessage` annotation contract (see realtime-channels.md). |
| GET | `/api/tasks/:id/debug/sizes` | **Debug-only.** Byte/token breakdown of the agent-facing `get_ticket` payload (measured off `?view=agent`), by section (`body`, `history`, `tags`, `cliSessions`, `frontmatter`) plus a history sub-breakdown (agent_session digests / comments / other). Optional `&historyLimit=N`. Powers the portal "Context budget" panel; the measurement is never attached to the agent surface itself. |
| GET | `/api/tasks/debug/mcp-schemas` | **Debug-only.** Spawns **every** MCP server an agent gets — EH module servers **and** the workspace `.mcp.json` servers (incl. event-horizon itself), deduped by id — runs the handshake + `tools/list`, and measures per-server tool-schema **and** `instructions` cost. Supports stdio/http/sse transports. On-demand and slow (starts real servers); each is time-boxed + failure-isolated (`ok:false`). Each server reports `source` (module/host/module+host). |
| GET | `/api/tasks/:id/debug/budget` | **Debug-only.** Broader "where does the agent context go" view: the `get_ticket` payload (as above) **plus** the launch prompt EH builds for the ticket (broken into EH instructions / echoed body / module fragments) **plus** the fixed skill modules that load every session **plus** (FLUX-1376) `ehToolSchemas` — EH's own MCP tool schemas measured in-process (`probeSelfMcpSchema`, no subprocess/network hop), with a per-tool breakdown — rolled into `ehMeasurableTotalTokensEst`. Explicit `caveats` cover what the engine still cannot see (host system prompt, external MCP schemas — see `/debug/mcp-schemas`, session accumulation). Token counts are a chars/4 estimate. |
| POST | `/api/bulk-rename` | Rename a status or tag across every ticket. Body: `{ kind: 'status' \| 'tag', from, to }`. Mounted at top level, not under `/api/tasks`, but lives in the same module. |

> **Attach to branch / parent ticket** (board card right-click → *Attach to*, FLUX-516) has **no dedicated endpoint** — both reuse `PUT /api/tasks/:id`. Attach-to-branch sets `branch` (no worktree created; execution-root resolution by branch routes it into that branch's worktree if one exists). Attach-to-parent sets `parentId`; the PUT handler keeps the parent's `subtasks` array and the child's `parentId` in sync bidirectionally. Attach-to-worktree is `POST /api/tasks/:id/worktree/join`.

### CLI session sub-routes (mounted under `/api/tasks`)

From [`cli-session.ts`](../../../engine/src/routes/cli-session.ts).

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/tasks/:id/cli-session` | Most recent CLI session summary for a ticket. |

> **Board orchestrator sentinel (`__board__`, FLUX-604).** The `start` / `input` / `stop` routes and the `GET /api/tasks/:id/cli-session` + `/transcript` reads special-case the reserved `id` `__board__` (`BOARD_CONVERSATION_ID`) — a persistent board-scoped chat **not bound to any ticket**. `start` body is `{ appendPrompt (first message), framework?, model?, effortOverride?, permissionMode?, attachments? }` (defaults to `gated`); `input` body is `{ message, model?, effortOverride?, permissionMode?, attachments? }`. **FLUX-959:** `framework` (`claude` | `copilot` | `gemini`; same validation as the per-ticket `start` route) selects which CLI the board spawns — it's fixed for the session's life (a follow-up `input` always resumes the framework the conversation started with, never re-reads `framework` from the request). The board session spawns the selected CLI at the workspace root with the full MCP toolset (Claude) or the workspace's `.mcp.json` (Copilot/Gemini) and writes **no ticket history** (it has no `sessionHistoryEntry`); its durable record is the transcript at `<fluxDir>/transcripts/__board__.jsonl` — shared across frameworks, so switching frameworks mid-conversation still recovers prior dialogue via the cold-resume re-prime. Turns end `waiting-input`, so a follow-up `--resume`s the same CLI session (memory persists). Implemented by the generic core (`startBoardSession`/`sendBoardInput`/`makeBoardAdapter` in [`agents/board-core.ts`](../../../engine/src/agents/board-core.ts)) plus one `BoardSpec` per framework ([`agents/claude-board.ts`](../../../engine/src/agents/claude-board.ts), [`agents/copilot-board.ts`](../../../engine/src/agents/copilot-board.ts), [`agents/gemini-board.ts`](../../../engine/src/agents/gemini-board.ts)), resolved via `getBoardAdapter(framework?)` in [`agents/index.ts`](../../../engine/src/agents/index.ts).

> **Furnace-chat sentinel (`__furnace__`, FLUX-1209).** A second, independent virtual conversation for the Furnace Operator ("Smelter") persona chat, launched from the Furnace drawer / dock flyout — previously (FLUX-1175) this rode on `__board__` itself, relabeling and overriding the orchestrator's own tab/transcript; it now gets its own sentinel (`FURNACE_CONVERSATION_ID`, `agents/board.ts`) so the two never cross-write. Two differences from the board: (1) `personaId` defaults to `'smelter'` when the request omits one (so a plain composer send that cold-starts a fresh turn on `__furnace__` — e.g. after a prior turn went terminal — can't accidentally boot a generic, unpersona'd identity); (2) the board-only long-term-memory machinery (the FLUX-838 cold-resume transcript re-prime, the FLUX-659 triage digest prepended to every turn) is skipped for `__furnace__` — it gets its own durable, resumable transcript (`<fluxDir>/transcripts/__furnace__.jsonl`) via the same generalized `board-core.ts` adapter, just without the whole-board context injection. A journaled write (any human/agent-authored mutation) survives a lost race this way; an un-journaled raw file edit does not — it's simply superseded by the remote's version. `conflict` is therefore no longer produced by the periodic tick itself; it's still reachable via the manual `resolveConflicts()`/`/api/storage/resolve-conflicts` path for a worktree that's genuinely mid-merge on disk (e.g. left by an older engine version). The `diverged` shape is `{ state:'diverged', ahead, behind }` (FLUX-1232) — local and remote `flux-data` have both moved since their common ancestor, surfaced by the STARTUP pull (separate from the periodic CAS tick); offer the reset-to-remote escape hatch above, never auto-reset. The `error` shape is `{ state:'error', error, errorType:'network'\|'auth'\|'conflict'\|'unknown', remediation? }`; for `errorType:'auth'` a `remediation: { reason, commands[] }` payload carries the engine-owned fix steps (FLUX-895). The `protocol-mismatch` shape is `{ state:'protocol-mismatch', required, supported }` (FLUX-1426) — the flux-data store's `sync-protocol` marker file (a single integer at the store root, seeded/forward-bumped by `ensureSyncProtocolMarker` in `storage-sync.ts` — FLUX-1428 bumped `SUPPORTED_SYNC_PROTOCOL` to `2` for CAS+replay) is ahead of `SUPPORTED_SYNC_PROTOCOL` in `sync-watcher.ts`; `runSync()`/`resolveConflicts()` perform **zero** mutating git ops (no add/commit/push) while gated — the board stays usable locally, read-only on sync — and the state clears on its own once a tick observes the marker back within range (e.g. after an engine upgrade). |
| POST | `/api/sync-status/sync` | Force an immediate sync (the "Retry" action). Re-detects gh auth on the spot so a just-completed `gh auth login` takes effect immediately. |
| POST | `/api/sync-status/test-error` | Inject a fake error state — UI development only. |
| GET | `/api/sync-status/stream` | SSE stream of sync status changes. |

**Sync authentication (orphan mode, FLUX-895).** All background-sync git calls run **non-interactively** (`GIT_TERMINAL_PROMPT=0`, `GCM_INTERACTIVE=never`) so a missing/expired credential can never open a login popup — it becomes a catchable `errorType:'auth'` instead. When `gh` is authenticated, the sync injects gh's credential helper for `https://github.com` (via `GIT_CONFIG_*`), so push/fetch "just works" with no interaction; when it isn't, the `auth` status carries `remediation` (`gh auth login` → `gh auth setup-git`) which the portal renders as an actionable "GitHub sign-in needed" panel, and a persistent `GitHub sign-in needed` notification is raised. On an `auth` failure the retry timer backs off from 30s to a slow cadence; a successful sync clears both the indicator and the notification. See [`git-sync-env.ts`](../../../engine/src/git-sync-env.ts).

## Perf (`/api/perf`)

From [`routes/perf.ts`](../../../engine/src/routes/perf.ts). **FLUX-1129:** the foundation of the engine's perf-instrumentation layer — see [`engine/src/perf/`](../../../engine/src/perf/registry.ts) in the code map. Not workspace-scoped (it reports process-level request timings, not ticket data), so it works even with no workspace active.

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/perf` | Dumps the in-memory perf registry snapshot: `{ counters, histograms, uptimeSeconds, rss }`. `counters` is a name→count map; `histograms` is a name→`{ count, sum, max, p50, p95 }` map, one entry per `http.<METHOD> <route>` (route params normalized, e.g. `/api/tasks/:id`) recorded by the `requestTiming` middleware mounted in `index.ts`. A request over the slow threshold (default 200ms, override with env `EH_PERF_SLOW_REQ_MS`) also logs a `[warn]` line via `log.warn`. Every response additionally carries a `Server-Timing: app;dur=<ms>` header. The SSE routes (`/api/events`, `/api/sync-status/stream`) are excluded from both timing and slow-request logging (long-lived by design). **FLUX-1130:** `histograms['eventloop.p50' \| 'eventloop.p99' \| 'eventloop.max']` — event-loop delay, sampled every 5s via `perf_hooks.monitorEventLoopDelay()` in [`perf/event-loop-monitor.ts`](../../../engine/src/perf/event-loop-monitor.ts) and started at boot in `index.ts`; each window's max delay over the stall threshold (default 150ms, override with env `EH_PERF_LOOP_STALL_MS`) also logs a `[perf] event loop stalled <N>ms` line via `log.warn` (at most once per 5s window). **FLUX-1131:** `histograms['git.<verb>' \| 'gh.<verb>']` (e.g. `git.status`, `git.fetch`, `gh.pr`) — every hardened git/gh subprocess (`runGit()`/`runGh()`/`runHardened()` in [`git-exec.ts`](../../../engine/src/git-exec.ts)), keyed by the first non-flag arg, timed via `GitOperationEvent.durationMs` and fed in by [`perf/git-timing.ts`](../../../engine/src/perf/git-timing.ts)'s sink (installed at boot in `index.ts`, alongside — not instead of — S9's operation-telemetry sink; `setGitOperationSink()` is a multicast). A call over the slow threshold (default 2000ms, override with env `EH_PERF_SLOW_GIT_MS`) logs a `[perf] slow git: <redacted cmd> took <N>ms` line via `log.warn`. **FLUX-1132:** `histograms['store.loadTask' \| 'store.fullRescan']` — `store.loadTask` times every `loadTask()` call (one per ticket file read, whether triggered by boot, the file watcher, or a direct write-back); `store.fullRescan` times `initDir()` (the actual disk rescan) alone, fed via [`perf/rescan-timing.ts`](../../../engine/src/perf/rescan-timing.ts). A rescan over the slow threshold (default 1000ms, override with env `EH_PERF_SLOW_RESCAN_MS`) logs a `[perf] slow full rescan: <N>ms` line via `log.warn`. **FLUX-1202:** a single `loadTask()` call over its own slow threshold (default 500ms, override with env `EH_PERF_SLOW_LOAD_TASK_MS`) additionally logs a `[perf] slow loadTask: <path> took <N>ms` line via [`perf/load-task-timing.ts`](../../../engine/src/perf/load-task-timing.ts)'s `warnIfSlowLoadTask()` and broadcasts a `slow-load-task` `perf` SSE event carrying the offending `filePath` — the aggregate `store.loadTask` histogram alone can't say *which* ticket produced an outlier, so this names it directly. **FLUX-1184:** `histograms['store.workspaceActivation']` — `activateWorkspace()`'s own umbrella duration (the whole workspace switch: watchers, sync, group docs, member binding, ... which nests the `initDir()` call above), recorded under this *separate* metric rather than also feeding `store.fullRescan` — previously one activation contributed two samples to the same rescan metric, reading as if the store had been rescanned twice at boot. Over its own slow threshold (default 2000ms, override `EH_PERF_SLOW_ACTIVATION_MS`) it logs a distinct `[perf] slow workspace activation: <N>ms` line. Also since FLUX-1184, `startWatchers()`'s chokidar instances set `ignoreInitial: true` — `initDir()` already loaded every ticket/doc directly, so chokidar's own initial-scan replay of an `add` per pre-existing file (previously a redundant second load of the whole board plus a spurious watch-event burst right at boot) is suppressed; only genuine post-boot changes count now. `counters['store.watchEvents']` — one per file-watcher-triggered reload (`add`/`change` on a top-level ticket file in `startWatchers()`); a rolling 10s window over the storm threshold (default 50, override with env `EH_PERF_WATCH_STORM`) logs `[perf] watcher storm: <N> fs events in 10s` via [`perf/watch-storm.ts`](../../../engine/src/perf/watch-storm.ts) (at most once per window). `counters['sse.broadcast.<event>']` (e.g. `sse.broadcast.taskUpdated`) and `histograms['sse.broadcastFanout']` — one broadcast counter per SSE event name and a timing histogram for the per-broadcast client write loop, both from `broadcastEvent()` in [`events.ts`](../../../engine/src/events.ts); `counters['sse.clients']` is the connected-client gauge (incremented on connect, decremented on disconnect/dead-write prune). |

### Debugging performance — `window.__ehPerf`

**FLUX-1133:** the client-side counterpart, mirroring the shape above but for the portal's own refresh pipeline. See [`portal/src/perfClient.ts`](../../../portal/src/perfClient.ts). Always attached (a few KB), so during a lag episode you can open the devtools console and type `__ehPerf.snapshot()` to see `{ counters, histograms, slowEvents, uptimeMs }`:

- `counters['refresh.trigger.poll' | 'refresh.trigger.visibility' | 'refresh.trigger.sse']` — what caused each `fetchTasks` refresh (the 3s poll, a tab focus/visibility resync, or an SSE-pushed `open`/`taskUpdated`/`taskDeleted`).
- `counters['sse.event.<type>']` — one counter per named SSE event received on the shared `/api/events` connection (`ping`, `activity`, `taskUpdated`, …) — useful for spotting which channel is flooding.
- `histograms['refresh.fetchTasks']` — timing for the `fetchTasks` call inside the main refresh, with the same `{ count, sum, max, p50, p95 }` shape as the engine registry.
- `slowEvents` — a ring buffer (last 100) of any `recordDuration` call over 300ms: `{ name, ms, at }`.

In dev (`import.meta.env.DEV`), a slow duration or an SSE burst (>30 events/5s) also logs a `[perf]`-prefixed `console.warn` as it happens, without needing to poll `snapshot()`.

**FLUX-1134:** rather than typing `__ehPerf.snapshot()` by hand, open the **Perf panel** — a hidden debug overlay that reads both snapshots (`GET /api/perf` + `window.__ehPerf.snapshot()`) and renders them side by side, no DevTools needed. It's a plain [`FloatingPanel`](../../../portal/src/components/FloatingPanel.tsx) (draggable/resizable, geometry persisted), so it can sit alongside the board while you reproduce a lag episode. See [`portal/src/components/PerfPanel.tsx`](../../../portal/src/components/PerfPanel.tsx).

- **Open it**: add `?perf=1` to the portal URL, or press **Alt+Shift+P** anywhere in the app (the shortcut also closes it again — it works with or without the query param, and the panel never appears in nav).
- **Sections**: HTTP (top routes by p95), Event loop (p50/p99/max stall), Git, Store / SSE, and Client — each is a dense `name / n / p50 / p95 / max` table. A row's `p95` over the matching threshold documented above (200ms HTTP, 150ms event loop, 2000ms git, 1000ms store rescan, 300ms client) is highlighted so the hot subsystem is visually obvious.
- **Git** and **Store / SSE** read the registry's `git.`/`gh.` and `store.`/`sse.` prefixed keys generically. Git timing landed in FLUX-1131; Store / SSE landed in FLUX-1132 — no panel changes were needed either way.
- Polls `GET /api/perf` every 3s while open (and only while open); the client side is read straight from `perfClient`'s `snapshot()` on the same tick — no extra subscription. The panel deliberately does **not** read `AppContext`/`useAppSelector`, so opening it never adds re-render load to the app it's measuring.
