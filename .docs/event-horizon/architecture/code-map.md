# Code Map

This page is the quick orientation guide for where behavior lives today — file by file. Pair it with [[Architecture Overview]] for the conceptual model and the subsystem table in the docs [[Documentation Index]].

## Product surfaces

-   `engine/src/packaged-mode.ts` is the **land-here-first** module for packaged-binary detection. Exports `isPkg` (true in `@yao-pkg/pkg` bundles), `isSea` (true in Node.js Single Executable Application mode), and `isPackaged` (either). In SEA mode, `ensureSeaAssetsExtracted()` reads the embedded asset manifest and writes all files (portal, skills, docs, tray binary, mcp-server.js) to a versioned directory under `os.tmpdir()` on first run; subsequent runs hit a `.extracted` marker and skip the copy. All other modules that need to distinguish "running as a binary" import from here rather than poking `process.pkg` directly.

-   `engine/src/index.ts` owns the main task, docs, and config API plus the repo watchers that keep the UI in sync with `.flux/` and `.docs/`. It also **mounts the Event Horizon MCP server in-process over loopback HTTP** (FLUX-645): `POST/GET/DELETE /mcp` → `handleMcpHttpRequest` (from `mcp-server.ts`), registered **before** `app.use(express.json())` so the raw JSON-RPC stream reaches the MCP transport unparsed (json would otherwise consume it and the transport hangs). `index.ts` **statically imports** `handleMcpHttpRequest`/`startMcpServer` from `mcp-server.ts` (FLUX-705) so esbuild bundles the MCP server into `index.js` and its tools run on the engine's **own** task-store instance (shared `workspaceRoot`/`tasksCache`/watchers). This replaced the pre-0.52 lazy `loadMcpModule()` that, in the packaged **SEA** build, `seaRequire`d `mcp-server.js` as a **separate bundle with its own, never-activated task-store** — so MCP writes threw `ERR_INVALID_ARG_TYPE: Received null` (`path.join` on a null `workspaceRoot`) and MCP reads were blind to tickets REST/portal had written, while REST itself worked. The trade-off is the MCP SDK now loads at engine startup instead of on the first `/mcp` hit. (The standalone `dist/mcp-server.js` bundle is still emitted but is now vestigial — see FLUX-705 follow-up.) `setEnginePort(PORT)` (from `packaged-mode.ts`) records the resolved listen port so the installer can render the loopback URL. The single-instance guarantee (EADDRINUSE refuses a second engine) makes the port unambiguous; the residual is that an engine restart drops every MCP connection (clients reconnect on the next call).

-   `engine/src/mcp-server.ts` is the **land-here-first** module for the **MCP tool surface**. `buildMcpServer()` constructs an `McpServer` and registers every ticket tool (`get_ticket`/`change_status`/`add_comment`/… ~27 tools) **without** connecting a transport — shared by both transports (FLUX-645). `startMcpServer()` is the headless `--mcp` stdio entry: it resolves the workspace (preferring `EH_CANONICAL_WORKSPACE`, else `resolveMainWorktree()` to redirect a linked worktree to the canonical store — FLUX-516/571, retained for stdio; broader fate in FLUX-646), `activateWorkspace`s, then `buildMcpServer().connect(new StdioServerTransport())`. `handleMcpHttpRequest(req,res)` is the in-process HTTP path mounted by `index.ts`: it keys a per-session `StreamableHTTPServerTransport` off the `Mcp-Session-Id` header (assigned by the transport on `initialize`, removed on close) so concurrent Claude sessions stay isolated, builds a fresh server per session, and never pre-parses the body. Tools operate on the engine's already-active task-store cache — guaranteed since FLUX-705 by bundling the MCP server into the engine bundle (see the `index.ts` entry above); destructive tools still route through the portal permission-prompt gate (`permission_prompt`). See [Reference: MCP Tools](../reference/mcp-tools.md).

-   `engine/src/workflow-installer.ts` owns workflow asset installation into a target repository. `buildMcpServerEntry()` renders the `event-horizon` `.mcp.json` entry — since FLUX-645 the **location-independent HTTP form** `{ type:"http", url:"http://127.0.0.1:<port>/mcp", alwaysLoad:true }` (no relative path, no `--workspace`, no worktree path), using `getEnginePort()`. It's re-written on every engine start, so a port change is picked up and a worktree session gets the same canonical URL as the main checkout.

-   `engine/src/modules.ts` is the **land-here-first** module for the **agent module system**. `loadModules()` reads `config.modules[]` and validates each entry (`isValidModule`). `getActiveModules(phase?, tags?)` filters by `enabled`, optional `phases` gating, and `conditions.requireTags`. `getModuleMcpServers(phase?, tags?)` returns active module MCP server declarations keyed by module id — consumed at agent spawn time by `claude-code.ts`; it resolves `${ACTIVE_FLUX_DIR}` in `mcpServer.env` values at call time so modules like `basic-memory` point at the workspace's active flux directory (`.flux-store` in orphan mode, `.flux` otherwise). `getModulePromptFragments(phase?, tags?)` builds the `## Active Modules` block injected into agent initial prompts (fragments capped at 2000 chars). `BUILTIN_MODULES` holds the catalog of built-in templates (Serena, Context7, Basic Memory, Mem0). A module may also declare `sharedHttp` (`{ command, args }` with `${PROJECT}`/`${PORT}` placeholders) — opting into an engine-managed shared HTTP server instead of a per-session stdio spawn (see `shared-mcp-server.ts`). Exposed to the portal via `GET /api/config/modules/catalog`.

-   `engine/src/module-probe.ts` is the **land-here-first** module for **MCP server operability checking**. `probeModule(m)` spawns the module's `mcpServer` process and settles after 5 s (still running → `ok`; clean exit → `ok`; non-zero exit → `error` with stderr). Results are stored in an in-memory map and broadcast via SSE `module-status` events so the portal updates live. `probeAllEnabled(modules)` probes all enabled modules in parallel — called at workspace activation and after a config save that enables new modules. `getProbeStatus(id)` / `getAllProbeStatuses()` are the read-side used by the config route and the `claude-code` adapter (which skips `error`-status servers at spawn time). For modules that declare `sharedHttp` on a proven platform, `probeModule` instead starts/reuses the shared server via `ensureSharedServer` and reports its HTTP health (no throwaway stdio spawn).

-   `engine/src/shared-mcp-server.ts` is the **land-here-first** module for **engine-managed shared MCP servers (FLUX-495)**. When a module opts in via `sharedHttp`, the engine runs ONE streamable-http server per project (Serena's language-server stack is expensive to spawn one-per-client) and hands every agent session the same `http://127.0.0.1:<port>/mcp` URL via `claude-code.ts`'s `buildModuleMcpConfigArgs`. `ensureSharedServer(m, projectPath)` allocates a free port (9200–9299), spawns the substituted launch command, and resolves once an MCP `initialize` handshake returns 200 — idempotent (reuses a healthy server) and concurrency-coalesced. `getSharedServerUrl(id)` is the sync read-side used at agent spawn. `shutdownSharedServers()` tree-kills every managed child (called from `gracefulShutdown` and `/api/shutdown`). Rollout is platform-gated by `isSharedHttpPlatformProven()` — on by default on Windows, fallback to the module's stdio `mcpServer` elsewhere; `EVENT_HORIZON_SHARED_MCP=0` forces stdio everywhere.

-   `engine/src/skill-installer.ts` is the CLI entrypoint used by the workspace `install-skill` command.

-   `engine/src/branch-manager.ts` owns all git plumbing for per-ticket branches: create, status, delete (local + remote), PR creation via `gh`, and the `finish_ticket` diff capture (numstat summary + unified-diff sidecar at `<flux-dir>/<ID>.diff`). `getPullRequestStatus(branch)` (FLUX-556) normalizes `gh pr view --json` into `PrStatus` (number/state/url/title/reviewDecision/mergeable + a `{total,passed,failed,pending}` check tally), returning null when no PR / gh unavailable — the read side of the in-EH PR card.

-   `engine/src/pr-tickets.ts` is the **land-here-first** module for **PR-as-a-first-class-ticket** (FLUX-566, epic FLUX-565). `syncPrTickets(workspaceRoot)` (wired into the same 90s poll as `reconcilePullRequests`) runs `gh pr list` and **upserts an engine-managed `PR-<n>` ticket** (`kind:'pr'`) per open PR via `task-store.upsertManagedTicket` (idempotent — writes only on a real change): it carries the PR metadata (`prNumber`/`prState`/`reviewDecision`/`isDraft`/`implementationLink`) and its **work-gated `members`** (normal tickets on the PR's branch that are In Progress/Ready — `selectMembers`, pure + tested). State mapping (decision #3) — `syncPrTickets` is the **PR-lifecycle owner** (FLUX-569): a NEW open PR → **Ready**; **CHANGES_REQUESTED → In Progress + `changes-requested` tint and its Ready members are bounced back to In Progress** (the review-fail unwind); a **reopened** PR (was Done / terminal `prState`, now OPEN again on gh) climbs back out of Done → **Ready** with `prState` reset to OPEN (fixes the FLUX-566 deferred edge); otherwise an existing open PR keeps its status (so a send-for-review move to In Progress isn't clobbered). PR tickets whose PR left the open list → **Done** (`prState` MERGED/CLOSED). `selectMembers` / `prTicketFields` / `prTicketId` are exported pure helpers (`pr-tickets.test.ts`). The synthetic-ticket write primitive `upsertManagedTicket` lives in `task-store.ts`. **Board rendering — the deck (FLUX-567, P2):** `portal/src/components/PrDeckCard.tsx` is the distinct PR card (violet theme, stacked-card shadow, state/review chips, changes-requested tint) — `Column.tsx`'s `renderTask` routes `kind:'pr'` tickets to it (grouped in the **Open PRs** section), everything else to `TaskCard`. `Board.tsx` does **deck-replace**: it excludes folded member ids (union of every PR ticket's `members`) from the per-column task lists, so members live *inside* the deck (unwind to reveal them with their real status), not in their own columns. A non-PR ticket whose `branch` is in `AppContext.prByBranch` (a PR's branch) but isn't folded — a Todo/Grooming/Backlog "pile" ticket — gets a `→ PR-n` marker in place (decision #4: never leaves its column). **Surface + actions (FLUX-568, P3):** `PrDeckSection`'s action bar is **state-aware** — when changes were requested (`swimlane === 'changes-requested'`) **Continue development** leads (launches a dev agent in the *implementation* phase on the PR branch) and Merge drops to secondary; otherwise **Review** (implementation→review launcher) + Merge (primary). Both route through the controller's `openLauncherInPhase(phase, templateId?)` so the launcher phase is set deterministically. Members stay folded during the fix loop (Review → changes-requested → Continue development → push → re-review → Merge). `TaskModalPopupView` renders `PrPanel` (full diff/status/send-for-review/update-branch) for any `kind:'pr'` ticket regardless of status. **Generalized deck — epics fold their subtasks (FLUX-580):** the fold/unwind UI is extracted into `portal/src/components/TaskDeck.tsx` (a `{ id, items, label, accent: 'violet'|'indigo' }` primitive — the `Layers` toggle + compact `hideStatusBadge` member cards, default folded) and now backs BOTH the PR deck (`PrDeckSection` renders `<TaskDeck accent="violet">`) and the **epic → subtasks** deck (`TaskCard`'s epic branch renders `<TaskDeck accent="indigo">` below `CardSubtaskProgress`). The shared fold rule lives in `portal/src/lib/decks.ts` (single source of truth, so Board and the card never drift): `collectPrMemberIds` (PR members), `isFoldedIntoEpic`/`epicDeckSubtasks` (a subtask folds into its epic **only when same status/column** AND not already a PR member — **PR precedence**), and `collectEpicFoldedIds` (the board-wide union). `Board.tsx`'s deck-replace now excludes `foldedMemberIds ∪ epicFoldedIds` from the columns; `useTaskCardController` exposes `epicFoldedSubtasks` and `AppContext`/`appStore` carry a memoized `prMemberIds` set so the card applies PR precedence without re-scanning every PR. Cross-column subtasks stay in their own column (unchanged).

-   `portal/src/components/EpicsScreen.tsx` is the **land-here-first** component for the **epic / roadmap view** (FLUX-678) — a top-level `AppView` (`'epics'`, registered like the other screens: `appStore.ts` union + `AppContext.tsx` `VIEW_PATHS`/`getViewFromLocation` (`/epics`) + `App.tsx` switch branch + `Header.tsx` `Target` NavItem). It lists every epic (`isEpic` = task with ≥1 subtask) with its count-based completion rate, sorted by % complete ascending, and drills in to the **remaining (non-done) subtasks grouped by board status** (group order derived from `config.columns`), every row clickable → `openTaskFullView`/`openTaskModal`. The completion math is **shared, not recomputed**: `portal/src/lib/epics.ts` (`isEpic` / `getDoneStatuses(config)` → {Done, Released, archive status} / `computeEpicRollup(epic, taskById, doneStatuses)` → `{ resolvedSubtasks, done, total, pct }`) is the single source of truth, consumed by BOTH this screen and `useTaskCardController` (board card), so done/total can never drift between the two surfaces. The progress bar primitive is `portal/src/components/EpicProgressBar.tsx`, extracted from `task-card/CardSubtaskProgress.tsx` so both render the identical bar. Phase 1 = list + drill-in only; a dated timeline/Gantt (`targetDate`) is deferred Phase 2.

-   **Onboarding "What you can do" feature panels (FLUX-755)** are **data-driven from a committed JSON config** with a hidden, dev-only editor. `portal/src/config/onboardingFeatures.json` is the **single committed seed** (`{ version, features: { id, icon, title, desc }[] }`); `portal/src/config/onboardingFeatures.ts` is the **land-here-first** typed loader — it statically imports the JSON (needs `resolveJsonModule`, now set in `portal/tsconfig.app.json`), declares `FeaturePanel`/`OnboardingFeaturesConfig`, and exports `validateFeatures()` (defensive coercion: drops non-objects, coerces fields to strings, auto-fills missing/duplicate ids, tolerates unknown icon names) + `FEATURE_PANELS`. `portal/src/config/featureIcons.ts` is the **single icon resolution path** — `resolveFeatureIcon(name)` over lucide-react's `icons` record (NO `DynamicIcon` in 1.14.0) with a `Sparkles` fallback, plus the curated `FEATURE_ICON_NAMES` list backing the picker. `portal/src/components/onboarding/FeatureHighlights.tsx` is the **sole renderer** of the feature-card grid (markup copied verbatim from the wizard's old inline grid, keyed by `id`, icon via `resolveFeatureIcon`) — reused by BOTH `OnboardingWizard.tsx` step 7 and the dev preview, so they are pixel-identical by construction. `portal/src/components/dev/OnboardingEditorScreen.tsx` is the **dev-only `/dev/onboarding` editor** (a `'dev-onboarding'` `AppView` registered like the other screens — `appStore.ts` union + `AppContext.tsx` `VIEW_PATHS`/`getViewFromLocation` + `Header.tsx` `NAV_TINTS`/NavItem + `App.tsx` render — but every render/route/nav site is **`import.meta.env.DEV`-gated** so the screen, its `lazy` import, and its dnd-kit usage are tree-shaken from the prod bundle and a hand-typed `/dev/onboarding` falls through to the board): LEFT = dnd-kit reorder (ChatDock pattern, vertical, id-keyed) + per-row icon picker + title/desc + add/remove + preview step/theme/width controls + Load/Save + unsaved badge; RIGHT = inline step-7 chrome around the shared `<FeatureHighlights features={draft} />` + a disabled no-op Continue (NO `OnboardingWizard` is embedded — zero wizard side-effect surface). The editor hydrates `draft` from the engine GET (the live file, fallback `FEATURE_PANELS`) and persists via PUT; after a save Vite HMR re-imports the changed JSON so an open wizard also refreshes. **Reachability gate (FLUX-755):** `App.tsx`'s `showOnboarding` (uncompleted onboarding → `<OnboardingWizard/>`) and the no-workspace `<WorkspaceSelector/>` early returns BOTH skip when `import.meta.env.DEV && view === 'dev-onboarding'` — without this the editor would be unreachable for the exact dev who clears `eh-onboarding-complete` to re-test the wizard (the early return would otherwise win before the Header nav link or the view branch ever render, and a hand-typed `/dev/onboarding` would be short-circuited to the wizard). The exception is dev-gated, so it cannot widen either gate in a prod build. **Engine side:** `engine/src/routes/dev-onboarding.ts` is a **dev-only** Router (`GET`/`PUT /api/dev/onboarding-features`) over the ONE hardcoded `PANELS_FILE` (`resolveSkillSourceRoot()` + `portal/src/config/onboardingFeatures.json`, NO request interpolation) with `isPathInsideRoot` + basename guards (403), PUT body-shape validation (400, write nothing), and atomic temp-write+rename. It is gated on TWO independent layers — `engine/src/index.ts` mounts it (WITHOUT `requireWorkspace`, since the file is repo-relative not workspace-relative) only when `DEV = !isPackaged && NODE_ENV !== 'production'`, and each handler re-checks `DEV` and 404s as a backstop. The portal API client adds `fetchOnboardingFeatures()`/`saveOnboardingFeatures()` in `portal/src/api.ts`. **These dev endpoints are intentionally NOT added to `.docs/event-horizon/reference/rest-api.md`** because they never ship to production (no packaged build mounts them).

-   **Data-driven onboarding flow (FLUX-756, Phase 1)** turns the WHOLE onboarding wizard data-driven: it renders from a committed flow config instead of a hardcoded `step===N` switch, preserving today's 9-page behavior byte-for-byte. `portal/src/config/onboardingFlow.json` is the **committed seed** (`{ version: 2, pages: OnboardingPage[] }` — a 1:1 transcription of the old 9 steps, copy verbatim). `portal/src/config/onboardingFlow.ts` is the **land-here-first** typed loader for the flow — it statically imports the JSON, declares the flow types (`OnboardingPage`/`OnboardingFlowConfig`/`OnboardingPageKind`/`OnboardingWidgetId`/`OnboardingCondition`/`OnboardingImage`/`OnboardingFeaturesRef`/`OnboardingCta`), and exports `SYSTEM_PAGE_SPECS` (the **single source of truth** for the 7 functional-step rails — `system`/`required`/`locked`/`dependsOn` per widget id), `DEFAULT_FLOW` (the seed mirrored in code), `validateFlow()`, and `FLOW = validateFlow(rawFlow)`. `validateFlow` mirrors `validateFeatures`' defensive posture (drop non-objects, coerce id/title to strings, auto-fill blank/dup ids as `page-<idx>`, default `kind` to `'content'` when `widget` is absent/unknown) and adds the **functional-step backstop**: it MERGES the matching `SYSTEM_PAGE_SPECS` row onto every `kind:'widget'` page (a hand-edit can never strip `required`/`locked`/`dependsOn`/`system`), runs a **topological check** (every widget page's `dependsOn` predecessors must appear earlier — on violation it re-derives the canonical SYSTEM ordering while keeping content pages in their authored slots), **re-injects** any missing required system page from `DEFAULT_FLOW`, and falls back to `DEFAULT_FLOW` on a missing/garbage file (incl. a legacy `{version,features}` shape with no `pages[]`). So a corrupt/hand-edited flow can never boot a broken order (e.g. install before folder-pick) or drop the folder-picker/completion. The schema is **forward-looking**: `image`/`conditions`/`mandatory`/`hidden`/`features.layout:'pages'` are PARSED + CARRIED but **inert** in Phase 1 (Phases 2-4 add behavior additively); Phase 1 only ACTS on `kind`/`widget`/copy/`icon`/`ctas`/`features.layout:'grid'` and the rails flags. `portal/src/components/onboarding/OnboardingWizard.tsx` now renders from `FLOW`: `const [pageIndex,setPageIndex]` (was `step`) walks `pages[]`; the 7 functional steps (incl. the terminal `completion`) are in-component `WIDGET_RENDERERS` closures keyed by `OnboardingWidgetId` (each the old `step===N` JSX MOVED verbatim over the **same** shared state + handlers — NO state lift, NO per-widget file extraction — with `setStep(n)→onAdvance()` and the install-skip jump `setStep(5)→goTo('bootstrap')`), and the 2 content pages (`features`, `docs`) render through one generic `renderContentPage` + a `CONTENT_ACTIONS` registry (`advance`/`open-docs`/`first-ticket`/`open-group`). `features:{ref:'onboardingFeatures',layout:'grid'}` resolves to `<FeatureHighlights features={FEATURE_PANELS}/>` so the **FLUX-755 feature seed is reused with zero duplication / zero migration**. The PATH-info effect is re-keyed to fire on `page.widget==='path-setup'` (was `step===6`). **Editor compat is deliberately minimal:** the FLUX-755 stack (`onboardingFeatures.json/.ts`, `featureIcons.ts`, `FeatureHighlights.tsx`, `OnboardingEditorScreen.tsx`, `api.ts`) is left UNTOUCHED — the flow is a parallel, separately-loaded config that references the panels via `featuresRef`, so the dev editor keeps compiling and loading/saving the `{version,features}` schema unchanged; the full flow Studio editor is Phase 2. **Engine side:** `engine/src/routes/dev-onboarding.ts` is left **UNTOUCHED** — its `validateConfigBody` still strictly validates ONLY the `{version,features:[{id,icon,title,desc}]}` shape and its `PANELS_FILE` write target stays `onboardingFeatures.json` (no flow-body branch is added, since the route can only safely persist the file it targets). **A dedicated GET/PUT `/api/dev/onboarding-flow` route (its own `onboardingFlow.json` target + `validateFlowBody`, cloned from this route's DEV gate + `isTargetSafe` + atomic temp+rename pattern) + `api.ts` fetch/save are NOT built (deferred to Phase 2)** — the wizard reads the flow via a static import, like `FEATURE_PANELS`, so no engine round-trip is needed at runtime. Both `OnboardingEditorScreen` and the `onboarding-features` API strings remain tree-shaken from the prod bundle (the whole dev editor is still `import.meta.env.DEV`-gated).

-   **Onboarding Studio (FLUX-759, Phase 2)** is the **land-here-first** surface for **editing the onboarding FLOW** (the per-page order/copy/widgets) — it is the Phase-2 flow editor the FLUX-756 entry above deferred, and it delivers the `/api/dev/onboarding-flow` route + `api.ts` fetch/save that FLUX-756 marked "not built". It **EXTENDS the existing dev-only `/dev/onboarding` surface** rather than adding a new view/route/nav seam: the same `'dev-onboarding'` `AppView`, `/dev/onboarding` route, `App.tsx` render branch + DEV-gated `lazy()` import, and Header nav link are reused unchanged (all four portal gates and the FLUX-755 prod-strip carry over). `portal/src/components/dev/OnboardingStudioScreen.tsx` is the **Studio shell** (the new single `lazy()` entry App.tsx points at, replacing the old `OnboardingEditorScreen`): a two-tab strip — **Flow** (default) and **Features** — driven by local `useState<'flow'|'features'>` (NO router change; URL stays `/dev/onboarding`). `portal/src/components/dev/FlowEditorTab.tsx` is the Flow tab: LEFT a per-page editor list (dnd-kit reorder reusing the Features-tab DndContext setup, keyed by `page.id`) with per-page title/subtitle/body/icon/ctas/features-grid-toggle/mandatory/hidden editors and **read-only system-rail chips** (`system`/`required`/`locked`/`depends on:`) for `kind:'widget'` pages; RIGHT the PLAY preview. It hydrates via `fetchOnboardingFlow()` (fallback to the imported `FLOW` seed), tracks `draft`/`lastSaved`/dirty, and Save persists `validateFlow(draft)` via `saveOnboardingFlow()`. **Reorder/delete UX is enforced from `SYSTEM_PAGE_SPECS`** (validateFlow-merged flags): required system pages (`pick-folder`/`storage-mode`/`pick-assistant`/`completion`) cannot be deleted (lock glyph), and `onDragEnd` runs the **exported** `isTopologicallyValid` predicate (the FLUX-759 Phase-1 change: added the `export` keyword to the formerly-private `isTopologicallyValid` in `portal/src/config/onboardingFlow.ts` — the ONLY edit to that file, so the client drag guard uses the SAME predicate the validator uses) against the proposed `arrayMove`, snapping back illegal drops with an inline reason; content pages move freely. `portal/src/components/dev/FlowPlayPreview.tsx` is the **play preview** — a Back/Next walker over `validateFlow(draft)` with reimplemented wizard chrome (StepDots + `renderContentPage` copied 1:1 from `OnboardingWizard.tsx`, class strings verbatim) that **NEVER imports `OnboardingWizard.tsx`** (HARD CONSTRAINT — FLUX-758 owns it; the source lines are cited in a file-header comment for drift auditing). Content pages reuse the SHARED `<FeatureHighlights features={FEATURE_PANELS}/>` (static seed, matching production — the Flow tab does NOT edit feature cards; that's the Features tab) with **inert no-op CTAs**; **widget/system pages render EXCLUSIVELY as inert amber dashed SANDBOX cards** showing the literal monospace banner `[system step: <widget> — runs live in real onboarding]` + a disabled Continue — markup-only, mounting NO widget renderer and firing ZERO side effects (no `pickWorkspaceFolder`/`installWorkspaceSkill`/`migrateStorage`/`setWorkspace`/`complete`), so a dev preview physically cannot mutate the user's real workspace; hidden pages still render but carry a Phase-4 badge. `portal/src/components/dev/FeaturesEditorTab.tsx` is the **Features tab** — the former `OnboardingEditorScreen` body extracted **byte-identical** (FLUX-755 unchanged), sharing the extracted `portal/src/components/dev/IconPicker.tsx` (`IconPicker` + `IconGlyph`, moved verbatim) with the Flow tab's per-page icon picker. The old `OnboardingEditorScreen.tsx` is **deleted** (its sole consumer was App.tsx, now repointed). **Engine side:** `engine/src/routes/dev-onboarding-flow.ts` is a **dev-only** Router (`GET`/`PUT /api/dev/onboarding-flow`) — a near-verbatim clone of `dev-onboarding.ts` targeting `portal/src/config/onboardingFlow.json` (`resolveSkillSourceRoot()` + path-guard `isTargetSafe`, NO request interpolation). PUT runs a **strict structural `validateFlowBody` pre-check** (non-array object + finite `version` + `pages` array + every page a `{string id, string title, kind in widget|content, known OnboardingWidgetId when widget}` object) → 400 + write nothing on malformed, else atomic temp-write+rename of the **RAW validated-shape body**. It deliberately does **NOT** import the portal's `validateFlow` as its validator — `validateFlow` never throws and silently coerces garbage to `DEFAULT_FLOW`, so using it would turn invalid input into a successful data-loss write; canonicalization (SYSTEM_PAGE_SPECS merge / topo re-derive / required re-injection) is the PORTAL's job at load + the client's pre-save `validateFlow`, and the engine guard's job is only to refuse structurally-broken JSON (documented in a route comment). Gated on the SAME two layers as the features route — `engine/src/index.ts` mounts it inside the existing `if (DEV)` block (WITHOUT `requireWorkspace`) beside the features router, and the handlers re-check `DEV` and 404 as a backstop. `portal/src/api.ts` adds `fetchOnboardingFlow()`/`saveOnboardingFlow()` (referenced ONLY from the dev Flow tab so they tree-shake from prod). **Prod-strip verified like FLUX-755:** a `npm run build -w portal` emits NO separate Studio async chunk and `portal/dist/assets` has ZERO matches for the dev sentinels (`[system step:`, `onboarding-flow`, `FlowPlayPreview`, `OnboardingStudio`, `FlowEditorTab`, `FeaturesEditorTab`, …) — the only residual `dev/onboarding` string is the pre-existing `VIEW_PATHS` route literal. **These dev endpoints are intentionally NOT added to `.docs/event-horizon/reference/rest-api.md`** for the same reason as the features route — no packaged build mounts them (`isPackaged` true).

-   **Substrate vs projection (FLUX-658).** `engine/src/transcript.ts` is the immutable, append-only **RAW substrate of record** — one JSONL file per stream at `<fluxDir>/transcripts/<streamId>.jsonl`. Since FLUX-658 each appended line is a **turn envelope** `{ v, turnId, streamId, seq, ts, role, raw }`: `raw` is the original stream-json / synthetic event stored verbatim, wrapped with a monotonic per-stream `seq` (== line position, seeded by `countLines` on first append so it survives restarts and continues past legacy lines) and `turnId = ${streamId}:${seq}`. This makes turns **addressable + sliceable** via `readTurns(streamId)` and `sliceTurns(streamId, fromSeq, toSeq)` (inclusive, open-ended bounds) — the primitives the future curation verbs (extract/merge) will call. Legacy pre-envelope lines read losslessly: `readTurns` wraps them on the fly (`seq` = line index), discriminating an envelope from a bare event by the `turnId`+`seq`+`raw` trio. The board **view** is a re-derivable **projection** of the substrate, not an independent store: `engine/src/projection.ts` owns the pure `projectTranscript(turns, ops?)` (no IO; owns the `Turn`/`TranscriptMessage` types + `classifyRole`), and `readTranscriptMessages` is now a thin `readTurns → projectTranscript` adapter (no behaviour change). The trailing `ops` param is the **seam** for the append-only curation op-log (`view = project(substrate, ops)`). **FLUX-656 ships the op-log STORE and the first verb:** `engine/src/curation-ops.ts` is the shared append-only log at `<fluxDir>/transcripts/_curation-ops.jsonl` (`appendCurationOp`/`readCurationOps`, write-queue serialized like the substrate), and `engine/src/extract.ts`'s `extractTicket(opts)` is the **promotion gate** — carve a `[fromSeq..toSeq]` slice of a source stream (default `__board__`) into a new card via one `extract` op, never copying or moving the source turns. The cross-stream gather lives in the reader (`transcript.ts`'s `gatherTurnsForView`: read ops → `sliceTurns` each `into===taskId` op → prepend, keeping foreign `streamId`) so `projectTranscript` stays pure; a `homeStreamId` arg tags foreign messages with `sourceStream`. `extractTicket` is the one path behind both the `extract_ticket` MCP tool (CONFIRM-gated) and the board-rebase `promote` executor. **FLUX-657 adds the inverse verb, `merge`:** `engine/src/merge.ts`'s `mergeTickets(opts)` folds several source streams into one survivor via a single `merge` op (`{ op:'merge', into, from:[…], by, ts }`) reusing the same op-log store; `gatherTurnsForView` honors `merge` ops by reading every `from` stream's turns and forming the **chronological union** (`ts` order, tie-break `(streamId, seq)`) with the survivor's own turns (foreign turns keep their `streamId` → `sourceStream` attribution). The sources are tombstoned (a `mergedInto` frontmatter pointer + pinned comment) and archived — never deleted — so removing the op reverts the fold. `mergeTickets` backs both the `merge_tickets` MCP tool (CONFIRM-gated) and the board-rebase `fold` executor. Design: [substrate-vs-projection.md](./substrate-vs-projection.md) §6.2–§6.3. Authored (kept verbatim) vs projected (re-derivable) field map lives in §7 of that doc.

-   `engine/src/hitl-prompts.ts` is the **land-here-first** module for the **restart-durable human-in-the-loop prompt store** (FLUX-833) shared by both HITL channels — gated-tool approvals (`permission-prompts.ts`, FLUX-605) and structured questions (`ask-questions.ts`, FLUX-662). Both channels park a single held `fetch` from the agent's MCP tool until a human resolves it (or it times out); their envelope is identical — `{ id, kind, payload, conversationId, claudeSessionId, createdAt }` — so they share this core, and the two original modules survive as **thin, name-preserving wrappers** (`requestApproval`/`requestAnswer`/`resolve*`/`listPending*` unchanged, so the HTTP routes + portal don't churn). The core: `parkPrompt(...)` mints a stable id, mirrors the open prompt to a durable index **`<fluxDir>/open-prompts.json`** (same active-flux-dir convention as `read-state.json`; the file is *exactly* the open set), appends the request transcript event, and returns the held Promise. The mirror write is **async + coalesced** (FLUX-854): `persist()` schedules an `fs.promises` write off the event loop and only ever keeps one in flight (concurrent park/settle calls mark the index dirty and the in-flight write re-loops to flush the latest snapshot), so a burst collapses to at most one extra write and **never** blocks the single-threaded loop the way the original synchronous `fs.*Sync` did — it ran on every park/settle and starved SSE + every portal API response (the FLUX-833 regression). Durability is *eventual*: `settle()` resolves the held fetch **before** scheduling the disk write (the up-front in-memory delete already makes it idempotent), and the exported `flushOpenPrompts(): Promise<void>` awaits the in-flight + pending write so tests can assert on-disk state deterministically. `settle(...)` is the **single funnel** for every resolution (human decision OR timeout) and is **idempotent** — the record is removed from the index up front, so a second settle (a late cross-restart answer racing a timeout that already wrote its `*-resolved`/`ask-answer`) is a no-op with **no phantom transcript entry** (the terminal-state guard; in-process this was previously covered only by `clearTimeout`). On the watcher `ready` hook (next to `reconcileOrphanedSessions` in `task-store.ts`), `rehydrateOpenPrompts()` reloads `open-prompts.json` and **re-broadcasts** each open prompt under its ORIGINAL id so the portal (SSE + the `listPending*` catch-up fetch) re-surfaces it — cards re-bind rather than orphan. The persisted `claudeSessionId` (captured from the live session by `index.ts`'s `resumePointerFor(conversationId)` at park time) is the future `claude --resume` target: a re-surfaced prompt's held fetch is dead after restart, so its answer can be recorded/dismissed but cannot yet reach the agent — delivering a post-restart answer to a live turn is the deliberately-gated **Phase 3** (`open-prompts.json` carries everything that phase needs). The kind-specific bits stay in the wrappers: permission = 120s snap → deny; question = 4 min deliberative → `unanswered` sentinel (held under undici's 300s `headersTimeout` so the long-poll fetch stays deliverable — see FLUX-662 review M1; lifting that ceiling is the independent Phase 4). Both channels still write their durable transcript events (`permission-request`/`-resolved`, `ask-question`/`ask-answer`) so a cold resume sees the round-trip (rendered by `projection.ts` — permission as quiet 🔒 note rows, questions as an assistant "❓" turn + a user "✔" turn). **Cross-ticket injection guard (FLUX-841):** the agent-supplied `conversationId` selecting the transcript stream is honored by the routes (`index.ts` `boundConversationId(req)`) only when the request also carries the requesting session's own **binding token** — `engine/src/session-binding.ts` mints `EH_CONVERSATION_TOKEN = signConversation(conversationId)` (HMAC of a per-process secret) into each session's env in `claude-code.ts`'s `cleanChildEnv`, the `permission_prompt`/`ask_user_question` MCP tools forward it, and a mismatched/forged id drops to `null` (unrouted). A session can only produce a valid token for its OWN ticket, so it can't append a HITL event into a sibling's transcript; `isSafeStreamId` (FLUX-833 M4) still independently blocks path traversal.

	in the renderer's kind-switch** — nothing is re-plumbed per surface. It also owns
	`buildStatusChangeHistory` / `changeTaskStatus`, the shared status-move history builder
	used by **both** `Board.tsx` and the registry so the `status_change` entry is built in
	one place. The imperative side lives in **`portal/src/hooks/useTicketActions.tsx`** (the
	host hook — status moves, agent dispatch, phase launches, PR merge, Return-to-dev, the
	launch-template catalog + the orchestration-launcher modal + Todo start-prompt state;
	this is the launch slice lifted out of `useTaskCardController`). **`portal/src/components/ticket-actions/TicketActions.tsx`**
	is the one renderer: `<TicketActions task variant="card"|"compact">` (self-contained:
	hook + buttons + launcher portals) for the chat surfaces, plus the lower-level
	`<TicketActionsView ctl variant>` (buttons only) and `<TicketActionsLaunchers ctl>`
	(launcher + start-prompt portals) that the **board card** drives from its controller's
	shared `useTicketActions` instance. It drives **three inline surfaces today** — the
	board card (`task-card/CardActionButtons.tsx` → thin `<TicketActionsView variant="card">`),
	the chat **mini-card** (`TaskMarkdown.tsx` `TicketChip` popover — whose old ▸ play button
	is removed; no more silent fire-and-forget launching), and the chat **composer bar**
	(`ChatPane` modal + `ChatDock` dock, via `ChatView`'s `actions` slot). The right-click
	`ContextMenu` + the `TaskModal` launch button are the **fast-follow** (subtask FLUX-717),
	designed to slot onto the same `category`/`kind` vocabulary without rework.

-   `portal/src/components/OrchestrationLauncher.tsx` is the **generic
	pattern-first launch modal**: pick an orchestration pattern (Scatter-gather /
	Parallel / Serialized / Hand-off), select participant roles from a catalog
	fetched from `GET /api/orchestration/personas?phase=<phase>` on open
	(phase-aware — grooming / implementation / review / finalize — and
	pattern-gated via `compatiblePatterns`), see a live `OrchestrationTopology`
	preview, and
	launch with partial-failure reporting. A **Template** dropdown lists every
	built-in and custom template defining a config for the current phase; selecting
	one re-applies its pattern + personas, and manual edits reset it to "Custom".
	On open it pre-populates from an `initialTemplateId` prop (the card's
	Single/Multi choice) or, failing that, the phase's single default
	(`resolvePhaseDefaultId(config.phaseDefaults, phase, 'single')`).
	When exactly **one** participant is selected the launcher treats it as a
	**standalone single agent**: the pattern selector is hidden and the run is
	always launchable (it bypasses orchestration gating). Consumers branch on
	`plan.personas.length === 1` → `runAgentAction({ action: { kind: 'persona', … } })`
	vs `launchOrchestration(...)` for teams, using `phaseCombiner(phase)` for the lead.
	Code review is one configuration of it.
	Claude Code is the only framework wired for now (no per-row framework picking);
	`serialized` / `handoff` are shown but gated (`launchable: false`) until the
	engine can sequence them. Opened from both `TaskModal` and `TaskCard`.

-   `portal/src/transcriptCache.ts` is the **land-here-first** module for **chat transcript render
	latency across minimize/reopen** (FLUX-750). A module-level LRU (`Map`, insertion order = recency,
	cap 12) keyed by `conversationId` with `getTranscript` / `hasTranscript` / `setTranscript`. The
	dock minimizes a chat by *unmounting* its `<ChatWindow>`, which destroys `useChatSession`'s local
	`messages` state — so a reopen used to cold-fetch from disk and flash blank. The hook now
	lazy-hydrates `messages` from this cache on mount (instant render, SWR-revalidated by the existing
	mount fetch) and writes through on every successful fetch; a true cache miss flips a `loading` flag
	that drives the cold-open spinner in `ChatView`. `ChatDock` keeps the cache fresh for a *running
	but minimized* conversation via a bounded warm subscription (only `running` ids not in `open`, no
	DOM, no per-idle-chat cost) so reopening a live session shows current committed state with no
	stale-then-jump. Pure module scope (no React) → survives unmount, resets on full reload — the same
	lifetime as `DockProvider`'s in-memory `drafts` map. `liveText` mid-stream tokens are intentionally
	NOT cached (the durable transcript via the warm-sub satisfies "current committed state immediately").

-   `portal/src/chatQueueCache.ts` is the sibling **land-here-first** module for the **FLUX-748 mid-turn
	message queue across minimize/reopen** (FLUX-837). Same lifetime trick as `transcriptCache` (module
	`Map` keyed by `conversationId`, survives the `<ChatWindow>` unmount, resets on full reload) but for
	the queue of messages the user submitted while a turn was `working`/`busy` — those live in
	`useChatSession`'s local `queued` state, so minimizing mid-turn used to *destroy an un-sent message*
	(it was queued, not yet in the durable transcript). The hook now lazy-hydrates `queued` from this
	cache, writes through on change, re-seeds on a conversationId switch, and — on a fresh reopen with a
	parked queue and a confirmed-idle session — dispatches the head once (the edge auto-dispatch can't
	fire on mount, so a turn that finished while minimized would otherwise leave the message parked).

-   `portal/src/components/task-modal/ChatView.tsx` owns **chat-open render latency** (FLUX-814) —
	complementary to `transcriptCache` (which fixes *reopen* latency). FLUX-750 made warm reopens
	instant, but a genuine **cold open** of a long thread still froze 3–4s: the engine transcript
	endpoint is cheap (~0.3s; the heavy `__board__` is a 7.8 MB / 6022-line `.jsonl` that projects to
	~803 messages / 242 KB), so the cost is the **frontend** synchronously mounting every message —
	each assistant turn a `TaskMarkdown` (react-markdown) parse. `ChatView` now **windows the render to
	the last `INITIAL_VISIBLE_MESSAGES` (80) turns** on open; a "Show earlier messages" pill reveals
	older turns in `LOAD_MORE_MESSAGES` (200) chunks with scroll position preserved (`restoreFromBottomRef`).
	The window grows with *incremental* live arrivals (gated `< INITIAL_VISIBLE_MESSAGES` so the one-shot
	bulk hydration on cold open does NOT expand it), collapses back to the tail on reset/conversation-switch,
	and **opens fully whenever find is open** (find walks the live DOM, so search must see the whole
	transcript). Absolute message indices are preserved through the window so the `data-last-msg` anchor
	(FLUX-727), unread divider (FLUX-695), and orchestration-block splice (FLUX-803) stay correct.

## Repo-backed data surfaces

-   `.flux/*.md` are the canonical ticket files.
    
-   `.flux/config.json` defines statuses, hidden statuses, tags, priorities, and workflow status names.
    
-   `.docs/**/*.md` are the project docs exposed inside the portal.
    
-   `.flux/skills/event-horizon-agent.md` is the source skill document.
    
-   `.flux/skills/event-horizon-copilot-instructions.md` is the source always-on Copilot instructions template.

-   `.flux-store/*.md` are the ticket files in **orphan-branch storage mode** — a git worktree on the `flux-data` branch that the engine auto-syncs (commit → fetch → merge → push) via `engine/src/sync-watcher.ts`. `runSync` is the **land-here-first** code for sync/merge behavior: it must **never `git add -A`/commit while a merge is unresolved** (MERGE_HEAD set or unmerged paths present) — doing so bakes git conflict markers into a ticket's YAML and corrupts it (FLUX-703/FLUX-694). On a conflicting merge it parks in `state:'conflict'` for user resolution (`resolveConflicts`); an in-memory `pendingConflicts` short-circuit plus an on-disk `hasUnmergedState` guard keep that safe across watcher self-triggers and engine restarts. A file that still ends up with committed `<<<<<<<`/`=======`/`>>>>>>>` markers is reported by `task-store.ts` `loadTask` as "unresolved git conflict markers" rather than a cryptic YAML error.
    

## Common change paths

-   Ticket schema or workflow behavior: start in `engine/src/index.ts`, then check `portal/src/types.ts`, `portal/src/api.ts`, and the relevant UI screen.

-   Docs experience changes: check `portal/src/components/DocsScreen.tsx`, `portal/src/components/DocsSidebar.tsx`, and the docs endpoints in `engine/src/index.ts`.
