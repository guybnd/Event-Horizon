# Code Map

This page is the quick orientation guide for where behavior lives today ŌĆö file by file. Pair it with [[Architecture Overview]] for the conceptual model and the subsystem table in the docs [[Documentation Index]].

## Product surfaces

-   `engine/src/packaged-mode.ts` is the **land-here-first** module for packaged-binary detection. Exports `isPkg` (true in `@yao-pkg/pkg` bundles), `isSea` (true in Node.js Single Executable Application mode), and `isPackaged` (either). In SEA mode, `ensureSeaAssetsExtracted()` reads the embedded asset manifest and writes all files (portal, skills, docs, tray binary, mcp-server.js) to a versioned directory under `os.tmpdir()` on first run; subsequent runs hit a `.extracted` marker and skip the copy. All other modules that need to distinguish "running as a binary" import from here rather than poking `process.pkg` directly.

-   `engine/src/index.ts` owns the main task, docs, and config API plus the repo watchers that keep the UI in sync with `.flux/` and `.docs/`. It also **mounts the Event Horizon MCP server in-process over loopback HTTP** (FLUX-645): `POST/GET/DELETE /mcp` ŌåÆ `handleMcpHttpRequest` (from `mcp-server.ts`), registered **before** `app.use(express.json())` so the raw JSON-RPC stream reaches the MCP transport unparsed (json would otherwise consume it and the transport hangs). `index.ts` **statically imports** `handleMcpHttpRequest`/`startMcpServer` from `mcp-server.ts` (FLUX-705) so esbuild bundles the MCP server into `index.js` and its tools run on the engine's **own** task-store instance (shared `workspaceRoot`/`tasksCache`/watchers). This replaced the pre-0.52 lazy `loadMcpModule()` that, in the packaged **SEA** build, `seaRequire`d `mcp-server.js` as a **separate bundle with its own, never-activated task-store** ŌĆö so MCP writes threw `ERR_INVALID_ARG_TYPE: Received null` (`path.join` on a null `workspaceRoot`) and MCP reads were blind to tickets REST/portal had written, while REST itself worked. The trade-off is the MCP SDK now loads at engine startup instead of on the first `/mcp` hit. (The standalone `dist/mcp-server.js` bundle is still emitted but is now vestigial ŌĆö see FLUX-705 follow-up.) `setEnginePort(PORT)` (from `packaged-mode.ts`) records the resolved listen port so the installer can render the loopback URL. The single-instance guarantee (EADDRINUSE refuses a second engine) makes the port unambiguous; the residual is that an engine restart drops every MCP connection (clients reconnect on the next call).

-   `engine/src/mcp-server.ts` is the **land-here-first** module for the **MCP tool surface**. `buildMcpServer()` constructs an `McpServer` and registers every ticket tool (`get_ticket`/`change_status`/`add_comment`/ŌĆ” ~27 tools) **without** connecting a transport ŌĆö shared by both transports (FLUX-645). `startMcpServer()` is the headless `--mcp` stdio entry: it resolves the workspace (preferring `EH_CANONICAL_WORKSPACE`, else `resolveMainWorktree()` to redirect a linked worktree to the canonical store ŌĆö FLUX-516/571, retained for stdio; broader fate in FLUX-646), `activateWorkspace`s, then `buildMcpServer().connect(new StdioServerTransport())`. `handleMcpHttpRequest(req,res)` is the in-process HTTP path mounted by `index.ts`: it keys a per-session `StreamableHTTPServerTransport` off the `Mcp-Session-Id` header (assigned by the transport on `initialize`, removed on close) so concurrent Claude sessions stay isolated, builds a fresh server per session, and never pre-parses the body. Tools operate on the engine's already-active task-store cache ŌĆö guaranteed since FLUX-705 by bundling the MCP server into the engine bundle (see the `index.ts` entry above); destructive tools still route through the portal permission-prompt gate (`permission_prompt`). See [Reference: MCP Tools](../reference/mcp-tools.md).

-   `engine/src/workflow-installer.ts` owns workflow asset installation into a target repository. `buildMcpServerEntry()` renders the `event-horizon` `.mcp.json` entry ŌĆö since FLUX-645 the **location-independent HTTP form** `{ type:"http", url:"http://127.0.0.1:<port>/mcp", alwaysLoad:true }` (no relative path, no `--workspace`, no worktree path), using `getEnginePort()`. It's re-written on every engine start, so a port change is picked up and a worktree session gets the same canonical URL as the main checkout.

-   `engine/src/modules.ts` is the **land-here-first** module for the **agent module system**. `loadModules()` reads `config.modules[]` and validates each entry (`isValidModule`). `getActiveModules(phase?, tags?)` filters by `enabled`, optional `phases` gating, and `conditions.requireTags`. `getModuleMcpServers(phase?, tags?)` returns active module MCP server declarations keyed by module id ŌĆö consumed at agent spawn time by `claude-code.ts`; it resolves `${ACTIVE_FLUX_DIR}` in `mcpServer.env` values at call time so modules like `basic-memory` point at the workspace's active flux directory (`.flux-store` in orphan mode, `.flux` otherwise). `getModulePromptFragments(phase?, tags?)` builds the `## Active Modules` block injected into agent initial prompts (fragments capped at 2000 chars). `BUILTIN_MODULES` holds the catalog of built-in templates (Serena, Context7, Basic Memory, Mem0). A module may also declare `sharedHttp` (`{ command, args }` with `${PROJECT}`/`${PORT}` placeholders) ŌĆö opting into an engine-managed shared HTTP server instead of a per-session stdio spawn (see `shared-mcp-server.ts`). Exposed to the portal via `GET /api/config/modules/catalog`.

-   `engine/src/module-probe.ts` is the **land-here-first** module for **MCP server operability checking**. `probeModule(m)` spawns the module's `mcpServer` process and settles after 5 s (still running ŌåÆ `ok`; clean exit ŌåÆ `ok`; non-zero exit ŌåÆ `error` with stderr). Results are stored in an in-memory map and broadcast via SSE `module-status` events so the portal updates live. `probeAllEnabled(modules)` probes all enabled modules in parallel ŌĆö called at workspace activation and after a config save that enables new modules. `getProbeStatus(id)` / `getAllProbeStatuses()` are the read-side used by the config route and the `claude-code` adapter (which skips `error`-status servers at spawn time). For modules that declare `sharedHttp` on a proven platform, `probeModule` instead starts/reuses the shared server via `ensureSharedServer` and reports its HTTP health (no throwaway stdio spawn).

-   `engine/src/shared-mcp-server.ts` is the **land-here-first** module for **engine-managed shared MCP servers (FLUX-495)**. When a module opts in via `sharedHttp`, the engine runs ONE streamable-http server per project (Serena's language-server stack is expensive to spawn one-per-client) and hands every agent session the same `http://127.0.0.1:<port>/mcp` URL via `claude-code.ts`'s `buildModuleMcpConfigArgs`. `ensureSharedServer(m, projectPath)` allocates a free port (9200ŌĆō9299), spawns the substituted launch command, and resolves once an MCP `initialize` handshake returns 200 ŌĆö idempotent (reuses a healthy server) and concurrency-coalesced. `getSharedServerUrl(id)` is the sync read-side used at agent spawn. `shutdownSharedServers()` tree-kills every managed child (called from `gracefulShutdown` and `/api/shutdown`). Rollout is platform-gated by `isSharedHttpPlatformProven()` ŌĆö on by default on Windows, fallback to the module's stdio `mcpServer` elsewhere; `EVENT_HORIZON_SHARED_MCP=0` forces stdio everywhere.

-   `engine/src/skill-installer.ts` is the CLI entrypoint used by the workspace `install-skill` command.

-   `engine/src/branch-manager.ts` owns all git plumbing for per-ticket branches: create, status, delete (local + remote), PR creation via `gh`, and the `finish_ticket` diff capture (numstat summary + unified-diff sidecar at `<flux-dir>/<ID>.diff`). `getPullRequestStatus(branch)` (FLUX-556) normalizes `gh pr view --json` into `PrStatus` (number/state/url/title/reviewDecision/mergeable + a `{total,passed,failed,pending}` check tally), returning null when no PR / gh unavailable ŌĆö the read side of the in-EH PR card.

-   `engine/src/pr-tickets.ts` is the **land-here-first** module for **PR-as-a-first-class-ticket** (FLUX-566, epic FLUX-565). `syncPrTickets(workspaceRoot)` (wired into the same 90s poll as `reconcilePullRequests`) runs `gh pr list` and **upserts an engine-managed `PR-<n>` ticket** (`kind:'pr'`) per open PR via `task-store.upsertManagedTicket` (idempotent ŌĆö writes only on a real change): it carries the PR metadata (`prNumber`/`prState`/`reviewDecision`/`isDraft`/`implementationLink`) and its **work-gated `members`** (normal tickets on the PR's branch that are In Progress/Ready ŌĆö `selectMembers`, pure + tested). State mapping (decision #3) ŌĆö `syncPrTickets` is the **PR-lifecycle owner** (FLUX-569): a NEW open PR ŌåÆ **Ready**; **CHANGES_REQUESTED ŌåÆ In Progress + `changes-requested` tint and its Ready members are bounced back to In Progress** (the review-fail unwind); a **reopened** PR (was Done / terminal `prState`, now OPEN again on gh) climbs back out of Done ŌåÆ **Ready** with `prState` reset to OPEN (fixes the FLUX-566 deferred edge); otherwise an existing open PR keeps its status (so a send-for-review move to In Progress isn't clobbered). PR tickets whose PR left the open list ŌåÆ **Done** (`prState` MERGED/CLOSED). `selectMembers` / `prTicketFields` / `prTicketId` are exported pure helpers (`pr-tickets.test.ts`). The synthetic-ticket write primitive `upsertManagedTicket` lives in `task-store.ts`. **Board rendering ŌĆö the deck (FLUX-567, P2):** `portal/src/components/PrDeckCard.tsx` is the distinct PR card (violet theme, stacked-card shadow, state/review chips, changes-requested tint) ŌĆö `Column.tsx`'s `renderTask` routes `kind:'pr'` tickets to it (grouped in the **Open PRs** section), everything else to `TaskCard`. `Board.tsx` does **deck-replace**: it excludes folded member ids (union of every PR ticket's `members`) from the per-column task lists, so members live *inside* the deck (unwind to reveal them with their real status), not in their own columns. A non-PR ticket whose `branch` is in `AppContext.prByBranch` (a PR's branch) but isn't folded ŌĆö a Todo/Grooming/Backlog "pile" ticket ŌĆö gets a `ŌåÆ PR-n` marker in place (decision #4: never leaves its column). **Surface + actions (FLUX-568, P3):** `PrDeckSection`'s action bar is **state-aware** ŌĆö when changes were requested (`swimlane === 'changes-requested'`) **Continue development** leads (launches a dev agent in the *implementation* phase on the PR branch) and Merge drops to secondary; otherwise **Review** (implementationŌåÆreview launcher) + Merge (primary). Both route through the controller's `openLauncherInPhase(phase, templateId?)` so the launcher phase is set deterministically. Members stay folded during the fix loop (Review ŌåÆ changes-requested ŌåÆ Continue development ŌåÆ push ŌåÆ re-review ŌåÆ Merge). `TaskModalPopupView` renders `PrPanel` (full diff/status/send-for-review/update-branch) for any `kind:'pr'` ticket regardless of status. **Generalized deck ŌĆö epics fold their subtasks (FLUX-580):** the fold/unwind UI is extracted into `portal/src/components/TaskDeck.tsx` (a `{ id, items, label, accent: 'violet'|'indigo' }` primitive ŌĆö the `Layers` toggle + compact `hideStatusBadge` member cards, default folded) and now backs BOTH the PR deck (`PrDeckSection` renders `<TaskDeck accent="violet">`) and the **epic ŌåÆ subtasks** deck (`TaskCard`'s epic branch renders `<TaskDeck accent="indigo">` below `CardSubtaskProgress`). The shared fold rule lives in `portal/src/lib/decks.ts` (single source of truth, so Board and the card never drift): `collectPrMemberIds` (PR members), `isFoldedIntoEpic`/`epicDeckSubtasks` (a subtask folds into its epic **only when same status/column** AND not already a PR member ŌĆö **PR precedence**), and `collectEpicFoldedIds` (the board-wide union). `Board.tsx`'s deck-replace now excludes `foldedMemberIds Ōł¬ epicFoldedIds` from the columns; `useTaskCardController` exposes `epicFoldedSubtasks` and `AppContext`/`appStore` carry a memoized `prMemberIds` set so the card applies PR precedence without re-scanning every PR. Cross-column subtasks stay in their own column (unchanged).

-   `portal/src/components/EpicsScreen.tsx` is the **land-here-first** component for the **epic / roadmap view** (FLUX-678) ŌĆö a top-level `AppView` (`'epics'`, registered like the other screens: `appStore.ts` union + `AppContext.tsx` `VIEW_PATHS`/`getViewFromLocation` (`/epics`) + `App.tsx` switch branch + `Header.tsx` `Target` NavItem). It lists every epic (`isEpic` = task with Ōēź1 subtask) with its count-based completion rate, sorted by % complete ascending, and drills in to the **remaining (non-done) subtasks grouped by board status** (group order derived from `config.columns`), every row clickable ŌåÆ `openTaskFullView`/`openTaskModal`. The completion math is **shared, not recomputed**: `portal/src/lib/epics.ts` (`isEpic` / `getDoneStatuses(config)` ŌåÆ {Done, Released, archive status} / `computeEpicRollup(epic, taskById, doneStatuses)` ŌåÆ `{ resolvedSubtasks, done, total, pct }`) is the single source of truth, consumed by BOTH this screen and `useTaskCardController` (board card), so done/total can never drift between the two surfaces. The progress bar primitive is `portal/src/components/EpicProgressBar.tsx`, extracted from `task-card/CardSubtaskProgress.tsx` so both render the identical bar. Phase 1 = list + drill-in only; a dated timeline/Gantt (`targetDate`) is deferred Phase 2.

-   `engine/src/pr-cleanup.ts` is the **land-here-first** module for the **in-EH PR merge ŌåÆ cleanup loop** (FLUX-557, epic FLUX-555). `cleanupMergedBranch(workspaceRoot, branch)` is the branch-scoped post-merge teardown: advance every non-Done ticket on the branch ŌåÆ Done, stop their sessions, fast-forward local master (`syncDefaultBranch`, FLUX-540 ŌĆö the engine reads the local tree, so a stale master makes diffs/collisions lie), then remove the worktree + force-delete the branch **only when the tree is clean**. A dirty worktree is kept and raises a persistent `info` notification with *Clean up worktree* / *Open worktree* actions (handled in `routes/notifications.ts`) ŌĆö work is never silently discarded, and the detach stash-to-master path is never used (the PR content already landed). The squash-merge gotcha: a `git log master..branch` "extra commits" gate would always false-positive (squash SHAs aren't ancestors of master), so gh's MERGED state is the source of truth and the dirty-tree check is the real safety gate. `reconcilePullRequests(workspaceRoot)` is the poll-based **boardŌåögh PR sync** (wired to a 90s interval in `index.ts`, gh-gated) ŌĆö the "EH gets notified about PRs" mechanism (decision #10: poll now, webhooks later). It now owns only the **out-of-band side-effects on a PR's *member* tickets + the worktree/branch** (the `PR-<n>` card itself is maintained by `syncPrTickets`). For **every non-terminal branch ticket** (deduped by branch ŌĆö **PR tickets `kind:'pr'` are excluded**; FLUX-598) it polls `gh` and reconciles: MERGED ŌåÆ `cleanupMergedBranch`; CLOSED ŌåÆ bounce **Ready members** back to In Progress + detach (abandon path; gated on `status===Ready` now the swimlane is gone, so it's idempotent); OPEN ŌåÆ **no-op for normal tickets** ŌĆö the PR's home is its `PR-<n>` deck card, so the FLUX-558 `open-pr` glow is **retired** (FLUX-569) and the path only mops up any legacy swimlane; no-PR ŌåÆ clear a stale `open-pr` swimlane. Quiet + idempotent ŌĆö writes only on a real change, never clobbers `require-input`. Called by `POST /api/tasks/:id/pr/merge` (`routes/tasks.ts`). `pruneMergedBranches(workspaceRoot)` (same 90s poll) is the **orphaned-branch backstop** (FLUX-599): the merge-time delete can be missed (branch checked out when cleanup ran, or all tickets already Done so reconcile skipped it ŌĆö incl. GitHub-direct merges), so it sweeps recently-merged PRs and force-deletes any `flux/` branch still present and not checked out / worktree-held. A branch it deems deletable whose delete keeps **throwing** (gh/remote outage, lock, perms) no longer fails silently: it raises a deduped `error` notification ("Couldn't clean up merged branch ŌĆ” ŌĆö delete it manually") **once per orphaned state** (in-module `notifiedStuckBranches` set, cleared when the branch leaves the existing set), so a genuinely-stuck branch surfaces instead of being retried forever in silence (FLUX-599). `branch-manager.deleteTicketBranch` now deletes local + remote **independently** (a checked-out local `-D` failure no longer skips the remote push), and `createPullRequest` only reuses an existing PR when its state is **OPEN** (a closed/merged PR on the branch no longer blocks opening a fresh one ŌĆö FLUX-597). **PR raise:** raising a PR (`POST /:id/pr`) stamps the PR link on every branch-sharing ticket ŌĆö it **no longer sets the `open-pr` swimlane** (retired, FLUX-569). **FLUX-558 UI retired (FLUX-569):** the `open-pr` swimlane/glow on *normal* tickets is gone ŌĆö `hasOpenPr`/`OPEN_PR_SWIMLANE` were removed from `portal/src/workflow.ts`, `TaskCard.tsx` dropped `pr-card-glow`, `Column.tsx`'s **Open PRs** group now holds only `kind:'pr'` deck cards, `TaskModalPopupView` renders `PrPanel` only for `kind:'pr'` tickets (the PR surface lives on the `PR-<n>` card), and `ContextMenu.tsx`'s **Raise PR** item shows for any branch ticket. The legacy `PrPanel.tsx` (live `fetchPrStatus` chips: state/checks/review/mergeable + Open-worktree / View-changes / Merge / Raise-PR) is now reached only through the PR ticket's modal. **Continue-development / unattached PRs (FLUX-569):** a zero-member PR card offers **Continue development** ŌåÆ `adoptPr` (`POST /:id/pr/adopt`, mode `adopt`|`create`) to bind an existing or fresh ticket to the branch so it folds into the deck; the **Merge** confirm lists the non-Done members it would sweep to Done and re-sends with `force` (the shared-PR guard). API client: `fetchPrStatus` / `raisePr` / `mergePr` (now takes `{force}`) / `adoptPr` in `portal/src/api.ts`. **Review flow (FLUX-559):** the PR card also surfaces stale (behind-count via `fetchBranchStatus`) + collision warnings (`fetchDiffOverview` `collidesWith`) with an **Update branch** action (`POST /:id/pr/update-branch` ŌåÆ merge default branch into the worktree, clean-tree-only, aborts on conflict) and a **Send for review** action that opens the review launcher (the existing reviewer-agent handoff bounces a failed review back to In Progress on the same worktree; the PR stays open and updates on push). **Unified finish path (FLUX-574):** `finish_ticket` (mcp-server) now routes its post-merge cleanup through `cleanupMergedBranch` too ŌĆö one ordering for both merge paths. `mergePullRequest` is **merge-only** (dropped `--delete-branch`, which threw *after* the merge landed when the branch was checked out in a worktree or the main tree); `cleanupMergedBranch` deletes the branch in the right order (remove worktree / switch the main tree off the branch ŌåÆ force-delete) and clears the `open-pr` swimlane (so finished tickets stop glowing as open PRs). **Final-message safety-net (FLUX-570):** `task-store.updateAgentSession` raises a notification when a session ends with a final message that reads like it needs input but the agent didn't route it to `require-input` ŌĆö so a blocking decision can't die silently in the session log. **finish self-sufficiency (FLUX-578):** finish_ticket ensures a PR exists before merging ŌĆö opens one if the Ready step didn't, or bounces with a "commit first" message when the branch has no commits. **Ticket delete teardown (FLUX-577):** `DELETE /api/tasks/:id` now detaches the ticket's own worktree (abandon path) + stops its sessions before unlinking, so deleting a ticket can't orphan its worktree.
    
-   `engine/src/group.ts` is the **land-here-first** module for multi-repo groups. It loads the committed `group.json` (machine-independent member identity) plus an optional gitignored `group.local.json` (per-machine paths), resolves member checkout paths (default `../<name>`), validates the config, and scaffolds the canonical `.flux-group` docs store. It also projects the group for agents/portal (`summarizeGroup`, with live `pathExists`) and builds the always-on sibling-source scope args (`buildMemberScopeArgs` ŌåÆ `--add-dir`, consumed by both agent adapters). For the **member side (FLUX-405, Case 1)** it owns the reverse-lookup binding: `activateMemberBinding()` scans the registered workspaces for a parent whose `group.json` lists this repo's `origin` remote ŌĆö compared via `normalizeRemoteForCompare()` (collapses https/ssh/scp spellings) and discovered without scaffolding via `peekGroupMembers()` ŌĆö and binds the parent's `GroupContext` (`getMemberBinding()`); a repo that is itself a parent never binds. The same reverse-lookup, applied across the **whole** registry, powers `resolveWorkspaceGroups(roots)` (FLUX-415) ŌĆö it tags each registered root as a group `parent`/`member` so `GET /api/workspaces` (`engine/src/routes/workspaces.ts`) can attach a `group` descriptor and the portal can render grouped repos nested together. `groupDocPathToStoreRelative(path, prefix)` maps a `<label>/<ŌĆ”>` doc path back to a store-relative file for edits (the prefix defaults to the active group's label). The docs label is **configurable per group** (`group.json` `docsLabel`, default `Product`) ŌĆö resolve it with `groupDocsLabel(group)` / `activeGroupDocsLabel()`; it's a display prefix only, so changing it never moves files (FLUX-414). `summarizeGroup` carries an optional `membership` descriptor (`{ role: 'parent' | 'member', groupName, parentRoot, memberName?, memberRole? }`, **FLUX-412**) plus `docsLabel`, attached by `GET /api/group` and the `get_project_group` MCP tool from `getGroupContext()` (parent) or `getMemberBinding()` (member) so a member workspace can show it belongs to a group while keeping `configured: false`. Purely additive ŌĆö inert when no `group.json` is present (single-repo mode unchanged). Activated from `activateWorkspace` in `task-store.ts`. See [Multi-Repo Groups](multi-repo-groups.md).
    
-   `engine/src/group-setup.ts` is the **land-here-first** module for *creating* a group (recreatability). `planGroupSetup()` computes every intrusive action (write `group.json`, patch `.gitignore`, scaffold the store, register/clone each member) with **zero git mutation** and returns a structured plan; `applyGroupSetup()` performs the writes only when asked, with per-member isolation and an injectable git runner for testing. `validateGitRemote()` screens every member `remote` before it reaches git (rejects shell metacharacters, `ext::`/`fd::`, `--upload-pack`/`--receive-pack`). It also owns the **Case-1 registration guardrail (FLUX-408)**: plan + apply register the dedicated parent and every present member as EH workspaces (injectable `listWorkspaces`/`registerWorkspace`, defaulting to `getWorkspacesList`/`addWorkspaceEntry`) so the member binding (FLUX-405) can reverse-look-up the parent; `ensureGroupRegistered(parentRoot, { dryRun })` is the idempotent **backfill** that brings an already-configured group's registry up to Case 1 without rewriting `group.json` (`dryRun` reports the gap without writing ŌĆö drives the detect-on-activation consent prompt). Surfaced headless via `engine/src/init-group.ts` (`npm run init-group`) and `engine/src/routes/group.ts` (`POST /api/group/plan` + `/apply` + `/ensure-registered`). The first slice registers existing checkouts; cloning is reported but not auto-performed. See [Multi-Repo Groups](multi-repo-groups.md).
    
-   `engine/src/group-discovery.ts` is the **land-here-first** module for the **onboarding/migration wizard (FLUX-407)** ŌĆö the read-only discovery + one-shot creation that the portal `GroupWizard` drives. `scanFolderForRepos(folder)` enumerates the *immediate* child git repos of a folder (no recursion, skips `node_modules`/`.git`/`.flux-group`/etc.), reading each repo's `origin` remote, whether it's already a registered EH workspace, and whether it's itself a group parent (via `peekGroupMembers`); `discoverFromRegistry()` projects the existing workspace registry the same way. `createDedicatedParent()` is the **dedicated-parent constructor**: it validates the group name + members (reusing `validateGitRemote` from `group-setup.ts`), **refuses to clobber an existing `group.json`**, then `mkdir`s the parent, `git init`s it (skipped if already a repo), scaffolds the `.flux-group` store, writes `group.json`, pins each member's discovered local `path` into a gitignored `group.local.json` (so the parent resolves members regardless of where it sits), registers the parent as an EH workspace labeled with the group name, and **registers every member whose checkout exists on disk** (labeled with the member name) ŌĆö returning a `memberRegistrations[]` report that flags any member with no supplied path or a path that isn't checked out (registration failures are non-fatal). Git + registry are injectable for testing. Surfaced via `engine/src/routes/group.ts` (`GET /api/group/discover/registry`, `POST /api/group/discover/folder`, `POST /api/group/create-parent`). Repairing/appending to an *already-configured* group routes through `ensureGroupRegistered` (FLUX-408) instead, since `createDedicatedParent` only creates new parents. See [Multi-Repo Groups](multi-repo-groups.md).
    
-   `engine/src/group-promote.ts` is the **land-here-first** module for **promoting existing `.docs/` into the group store (FLUX-404)** ŌĆö the bridge between a repo's main-branch docs and the `flux-group-docs` canonical store, with **move semantics** (mirrors the ticket-migration precedent in `storage-sync.ts`). `planDocsPromotion(parentRoot)` walks `.docs/` (zero mutation) and proposes a store-relative target per file (default `features/<basename>`, retargetable). `collectPromotions(sourceRoot, selections, storeDir?)` is the pure, unit-testable core: it validates every source is under `.docs/` and every target is inside the store (rejecting `..`/absolute/`.git`), reads content, and aborts the batch up front on a bad path ŌĆö `storeDir` defaults to the source repo's own store but a member passes the **parent's** store. `applyDocsPromotion(group, selections, { gitRunner? })` performs the **parent-origin** move ŌĆö write into the `.flux-group` worktree, `git rm` each source from main, commit the removals, then `syncGroup` to commit on `flux-group-docs` and fan out. `applyMemberDocsPromotion(memberRoot, parentGroup, selections, { gitRunner? })` performs the **member-origin** move (FLUX-406, push-through-parent): read the member's own `.docs/`, write the content into the store *through the parent* (`submitGroupEdit`), then `git rm` each source from the member's own main; the doc returns as a read-only group doc. 	`listSelectablePersonaMeta(phase?)`. Built-ins are **viewable but read-only**
	(`getEditablePersona` returns them so the UI can show the prompt and offer a
	fork); `saveCustomPersona` refuses ids that collide with a built-in, so the
	curated set stays owned by Event Horizon and is updated via app releases. The
	portal fetches metadata via `GET /api/orchestration/personas` and reads/edits
	full personas through the CRUD routes in `engine/src/routes/orchestration.ts`.

-   `engine/src/models/workflow.ts` owns reusable **workflow templates**
	(per-phase pattern + persona membership). It ships a code-defined roster
	(`BUILTIN_WORKFLOWS`, each `builtIn: true`): a single-agent and a multi-agent
	template per phase, keyed `builtin-<phase>-single` / `builtin-<phase>-multi`.
	Custom templates persist under `<fluxDir>/workflows/*.json`; `loadWorkflows`
	merges built-ins first, `isBuiltInWorkflow(id)` guards `saveWorkflow` /
	`deleteWorkflow` (built-ins are read-only, updated via releases ŌĆö duplicate to
	customize). Exposed through `engine/src/routes/workflows.ts` (`/api/workflows`;
	PUT/DELETE return 400 for built-in ids). Per-phase launch defaults are stored on
	`config.phaseDefaults[phase].single` / `.multi` (each a template id; falls back
	to `builtin-<phase>-<variant>`). The legacy board-wide `config.defaultWorkflowId`
	still exists but is superseded by the per-phase defaults.
    
-   `portal/src/App.tsx` wires the top-level screens.
    
-   `portal/src/AppContext.tsx` coordinates view state, routing-like context,
	and the shared live task polling plus change-event state used by board,
	backlog, and header surfaces.
    
-   `portal/src/api.ts` is the portal's client for engine endpoints.

-   `portal/src/agentActions.ts` is the **single composition layer** for
	launching agent CLI sessions. Every button that starts an agent (card
	context menu, card quick-actions, modal CLI panel, modal Finish/Grooming,
	Code Review picker) routes through `runAgentAction(...)`. For multi-agent
	runs, use the generic `launchOrchestration(...)` ŌĆö it takes a pattern-first
	`OrchestrationMode` (from `ORCHESTRATION_MODES`) plus ordered participants and
	an optional combiner/lead, and stamps the shared `groupId` + pattern metadata.
	Participants reference a persona by `personaId` (+ optional `focusComment`);
	prompt text is resolved server-side. `ReviewPersona` is metadata-only ŌĆö fetch
	the catalog with `fetchOrchestrationPersonas()`.
	`runParallelReviews(...)` / `launchOrchestratedReview(...)` are thin code-review
	presets over it. Add new launch entry points here, not by calling
	`startTaskCliSessionEx` directly.

-   `portal/src/lib/ticketActions.ts` is the **single source of truth for the
	phase-aware chat action bar** (FLUX-610): `actionsForStatus(task, ctx)` maps a
	ticket's status to the right `TicketAction[]`, split by `kind` ŌĆö `engine`
	(direct REST status move / merge, zero tokens), `agent` (tokenized dispatch via
	`agentActions`), `link` (open PR). It also owns `buildStatusChangeHistory` /
	`changeTaskStatus`, the shared status-move history builder used by **both**
	`Board.tsx` and the bar so the `status_change` entry is constructed in one
	place. `portal/src/components/TicketActionBar.tsx` renders it and is passed into
	the dumb `ChatView` via its `actions` slot by `ChatPane` (modal) and the dock
	`ChatWindow`.

-   `portal/src/components/OrchestrationLauncher.tsx` is the **generic
	pattern-first launch modal**: pick an orchestration pattern (Scatter-gather /
	Parallel / Serialized / Hand-off), select participant roles from a catalog
	fetched from `GET /api/orchestration/personas?phase=<phase>` on open
	(phase-aware ŌĆö grooming / implementation / review / finalize ŌĆö and
	pattern-gated via `compatiblePatterns`), see a live `OrchestrationTopology`
	preview, and
	launch with partial-failure reporting. A **Template** dropdown lists every
	built-in and custom template defining a config for the current phase; selecting
	one re-applies its pattern + personas, and manual edits reset it to "Custom".
	On open it pre-populates from an `initialTemplateId` prop (the card's
	Single/Multi choice) or, failing that, the phase's single default
	(`resolvePhaseDefaultId(config.phaseDefaults, phase, 'single')`).
	When exactly **one** participant is selected the launcher treats it as a
	**standalone single agent**: the pattern selector is hidden and the run is
	always launchable (it bypasses orchestration gating). Consumers branch on
	`plan.personas.length === 1` ŌåÆ `runAgentAction({ action: { kind: 'persona', ŌĆ” } })`
	vs `launchOrchestration(...)` for teams, using `phaseCombiner(phase)` for the lead.
	Code review is one configuration of it.
	Claude Code is the only framework wired for now (no per-row framework picking);
	`serialized` / `handoff` are shown but gated (`launchable: false`) until the
	engine can sequence them. Opened from both `TaskModal` and `TaskCard`.

-   `portal/src/components/WorkflowBuilder.tsx` is the **Workflows screen** (backed
	by `/api/orchestration/personas`, `/api/workflows`, and `/api/config`). Three
	tabs: **Personas** (per-phase; built-ins are viewable read-only with a
	"Duplicate & Edit" fork, custom personas create/edit/delete),
	**Templates** (grouped by phase, each phase split into Single / Multi columns by
	persona count; a per-column star sets `config.phaseDefaults[phase].single` /
	`.multi` via `handleSetPhaseDefault`; cards show resolved pattern + ordered
	persona chips), and **Skills** (skills persist as docs under the
	`skills/` directory).

-   `portal/src/components/CodeReviewButton.tsx` owns the entry button (compact +
	full variants) that opens the `OrchestrationLauncher`. Used by both `TaskModal`
	and `TaskCard`.

-   `portal/src/orchestration.ts` is the **land-here-first** module for
	multi-agent run logic on the portal side. It groups a ticket's
	`cliSessions[]` into runs by shared `groupId`, aggregates per-run status,
	and derives the topology shape / labels. Pure functions, no React. Paired
	with `portal/src/components/OrchestrationTopology.tsx`, which renders the
	run shape as a compact glyph (cards/popover) or a per-agent node map
	(modal Run View).

-   `portal/src/components/task-modal/RunView.tsx` is the modal surface for a
	live multi-agent run: topology map header, per-agent collapsible output,
	per-session and group-level Stop, and the scatter-gather barrier banner.
	Single-session tickets still use `task-modal/CliSessionPanel.tsx`;
	`TaskModal` switches to `RunView` when an active 2+ session group exists.

## Repo-backed data surfaces

-   `.flux/*.md` are the canonical ticket files.
    
-   `.flux/config.json` defines statuses, hidden statuses, tags, priorities, and workflow status names.
    
-   `.docs/**/*.md` are the project docs exposed inside the portal.
    
-   `.flux/skills/event-horizon-agent.md` is the source skill document.
    
-   `.flux/skills/event-horizon-copilot-instructions.md` is the source always-on Copilot instructions template.

-   `.flux-store/*.md` are the ticket files in **orphan-branch storage mode** ŌĆö a git worktree on the `flux-data` branch that the engine auto-syncs (commit ŌåÆ fetch ŌåÆ merge ŌåÆ push) via `engine/src/sync-watcher.ts`. `runSync` is the **land-here-first** code for sync/merge behavior: it must **never `git add -A`/commit while a merge is unresolved** (MERGE_HEAD set or unmerged paths present) ŌĆö doing so bakes git conflict markers into a ticket's YAML and corrupts it (FLUX-703/FLUX-694). On a conflicting merge it parks in `state:'conflict'` for user resolution (`resolveConflicts`); an in-memory `pendingConflicts` short-circuit plus an on-disk `hasUnmergedState` guard keep that safe across watcher self-triggers and engine restarts. A file that still ends up with committed `<<<<<<<`/`=======`/`>>>>>>>` markers is reported by `task-store.ts` `loadTask` as "unresolved git conflict markers" rather than a cryptic YAML error.
    

## Common change paths

-   Ticket schema or workflow behavior: start in `engine/src/index.ts`, then check `portal/src/types.ts`, `portal/src/api.ts`, and the relevant UI screen.
    
-   Prompt or review workflow changes: check `engine/src/orchestration-personas.ts` (reviewer/orchestrator prompts + resolver), `portal/src/agentActions.ts` (launch helper + command registry), `portal/src/components/TaskModal.tsx`, `portal/src/components/Settings.tsx`, `.flux/config.json`, and the workflow asset templates under `.flux/skills/`.
    
-   Docs experience changes: check `portal/src/components/DocsScreen.tsx`, `portal/src/components/DocsSidebar.tsx`, and the docs endpoints in `engine/src/index.ts`.
