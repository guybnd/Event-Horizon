# Code Map

This page is the quick orientation guide for where behavior lives today ‚Äî file by file. Pair it with [[Architecture Overview]] for the conceptual model and the subsystem table in the docs [[Documentation Index]].

## Product surfaces

-   `engine/src/packaged-mode.ts` is the **land-here-first** module for packaged-binary detection. Exports `isPkg` (true in `@yao-pkg/pkg` bundles), `isSea` (true in Node.js Single Executable Application mode), and `isPackaged` (either). In SEA mode, `ensureSeaAssetsExtracted()` reads the embedded asset manifest and writes all files (portal, skills, docs, tray binary, mcp-server.js) to a versioned directory under `os.tmpdir()` on first run; subsequent runs hit a `.extracted` marker and skip the copy. All other modules that need to distinguish "running as a binary" import from here rather than poking `process.pkg` directly.

-   `engine/src/index.ts` owns the main task, docs, and config API plus the repo watchers that keep the UI in sync with `.flux/` and `.docs/`. It also **mounts the Event Horizon MCP server in-process over loopback HTTP** (FLUX-645): `POST/GET/DELETE /mcp` ‚Üí `handleMcpHttpRequest` (from `mcp-server.ts`), registered **before** `app.use(express.json())` so the raw JSON-RPC stream reaches the MCP transport unparsed (json would otherwise consume it and the transport hangs). `index.ts` **statically imports** `handleMcpHttpRequest`/`startMcpServer` from `mcp-server.ts` (FLUX-705) so esbuild bundles the MCP server into `index.js` and its tools run on the engine's **own** task-store instance (shared `workspaceRoot`/`tasksCache`/watchers). This replaced the pre-0.52 lazy `loadMcpModule()` that, in the packaged **SEA** build, `seaRequire`d `mcp-server.js` as a **separate bundle with its own, never-activated task-store** ‚Äî so MCP writes threw `ERR_INVALID_ARG_TYPE: Received null` (`path.join` on a null `workspaceRoot`) and MCP reads were blind to tickets REST/portal had written, while REST itself worked. The trade-off is the MCP SDK now loads at engine startup instead of on the first `/mcp` hit. (The standalone `dist/mcp-server.js` bundle is still emitted but is now vestigial ‚Äî see FLUX-705 follow-up.) `setEnginePort(PORT)` (from `packaged-mode.ts`) records the resolved listen port so the installer can render the loopback URL. The single-instance guarantee (EADDRINUSE refuses a second engine) makes the port unambiguous; the residual is that an engine restart drops every MCP connection (clients reconnect on the next call).

-   `engine/src/mcp-server.ts` is the **land-here-first** module for the **MCP tool surface**. `buildMcpServer()` constructs an `McpServer` and registers every ticket tool (`get_ticket`/`change_status`/`add_note`/‚Ä¶ 24 tools after the FLUX-882 consolidation ‚Äî single-op tools folded behind `action`/`type` params; see the [migration map](../reference/mcp-tools.md#flux-882-tool-consolidation-migration)) **without** connecting a transport ‚Äî shared by both transports (FLUX-645). `startMcpServer()` is the headless `--mcp` stdio entry: it resolves the workspace (preferring `EH_CANONICAL_WORKSPACE`, else `resolveMainWorktree()` to redirect a linked worktree to the canonical store ‚Äî FLUX-516/571, retained for stdio; broader fate in FLUX-646), `activateWorkspace`s, then `buildMcpServer().connect(new StdioServerTransport())`. `handleMcpHttpRequest(req,res)` is the in-process HTTP path mounted by `index.ts`: it keys a per-session `StreamableHTTPServerTransport` off the `Mcp-Session-Id` header (assigned by the transport on `initialize`, removed on close) so concurrent Claude sessions stay isolated, builds a fresh server per session, and never pre-parses the body. Tools operate on the engine's already-active task-store cache ‚Äî guaranteed since FLUX-705 by bundling the MCP server into the engine bundle (see the `index.ts` entry above); destructive tools still route through the portal permission-prompt gate (`permission_prompt`). See [Reference: MCP Tools](../reference/mcp-tools.md).

-   `engine/src/log.ts` is the **stdout-safety guard** (FLUX-351): a dependency-free structured logger that writes **only to stderr** (`log.info`/`warn`/`error`/`debug`, each `process.stderr.write` with a level prefix). In stdio MCP mode the `StdioServerTransport` (`mcp-server.ts`) owns **stdout** for JSON-RPC framing, so any stray `console.log` (stdout) corrupts the protocol ‚Äî all diagnostic logging across `engine/src` routes through `log` instead, and `engine/eslint.config.js`'s `no-console` rule (allowing `warn`/`error`) bans `console.log` in engine source so the hazard can't be reintroduced. The deliberate transport stdout writes live in the SDK and are untouched.

-   `engine/src/workflow-installer.ts` owns workflow asset installation into a target repository. `buildMcpServerEntry()` renders the `event-horizon` `.mcp.json` entry ‚Äî since FLUX-645 the **location-independent HTTP form** `{ type:"http", url:"http://127.0.0.1:<port>/mcp", alwaysLoad:true }` (no relative path, no `--workspace`, no worktree path), using `getEnginePort()`. It's re-written on every engine start, so a port change is picked up and a worktree session gets the same canonical URL as the main checkout. **Forced reinstall / hard-override (FLUX-882, scoped in FLUX-942):** `installWorkspaceWorkflow({ force? })` writes the current skill surface, then calls `cleanOrphanedSkillFiles(targetDir, framework)` ‚Äî which deletes stale `event-horizon*` files inside **the resolved framework's own install dir(s)** (the dirnames of the files it writes, via `skillDestinationFor`/`skillModuleDestinationFor`) that the current install no longer writes, so a renamed/removed skill module from an older release can't linger and shadow the new surface. It is scoped to that one framework's dir ‚Äî **never other frameworks' dirs** ‚Äî so a multi-framework workspace (e.g. Claude + Gemini) keeps each framework's installed file (FLUX-942 fixed the original sweep, which scanned every framework's dir and deleted the others' files on every restart). The sweep is idempotent (best-effort per file) so it runs on every install; `force` logs it and guarantees it.

-   `engine/src/modules.ts` is the **land-here-first** module for the **agent module system**. `loadModules()` reads `config.modules[]` and validates each entry (`isValidModule`). `getActiveModules(phase?, tags?)` filters by `enabled`, optional `phases` gating, and `conditions.requireTags`. `getModuleMcpServers(phase?, tags?)` returns active module MCP server declarations keyed by module id ‚Äî consumed at agent spawn time by `claude-code.ts`; it resolves `${ACTIVE_FLUX_DIR}` in `mcpServer.env` values at call time so modules like `basic-memory` point at the workspace's active flux directory (`.flux-store` in orphan mode, `.flux` otherwise). `getModulePromptFragments(phase?, tags?)` builds the `## Active Modules` block injected into agent initial prompts (fragments capped at 2000 chars). `BUILTIN_MODULES` holds the catalog of built-in templates (Serena, Context7, Basic Memory, Mem0). A module may also declare `sharedHttp` (`{ command, args }` with `${PROJECT}`/`${PORT}` placeholders) ‚Äî opting into an engine-managed shared HTTP server instead of a per-session stdio spawn (see `shared-mcp-server.ts`). Exposed to the portal via `GET /api/config/modules/catalog`.

-   `engine/src/module-probe.ts` is the **land-here-first** module for **MCP server operability checking**. `probeModule(m)` spawns the module's `mcpServer` process and settles after 5 s (still running ‚Üí `ok`; clean exit ‚Üí `ok`; non-zero exit ‚Üí `error` with stderr). Results are stored in an in-memory map and broadcast via SSE `module-status` events so the portal updates live. `probeAllEnabled(modules)` probes all enabled modules in parallel ‚Äî called at workspace activation and after a config save that enables new modules. `getProbeStatus(id)` / `getAllProbeStatuses()` are the read-side used by the config route and the `claude-code` adapter (which skips `error`-status servers at spawn time). For modules that declare `sharedHttp` on a proven platform, `probeModule` instead starts/reuses the shared server via `ensureSharedServer` and reports its HTTP health (no throwaway stdio spawn).

-   `engine/src/shared-mcp-server.ts` is the **land-here-first** module for **engine-managed shared MCP servers (FLUX-495)**. When a module opts in via `sharedHttp`, the engine runs ONE streamable-http server per project (Serena's language-server stack is expensive to spawn one-per-client) and hands every agent session the same `http://127.0.0.1:<port>/mcp` URL via `claude-code.ts`'s `buildModuleMcpConfigArgs`. `ensureSharedServer(m, projectPath)` allocates a free port (9200‚Äì9299), spawns the substituted launch command, and resolves once an MCP `initialize` handshake returns 200 ‚Äî idempotent (reuses a healthy server) and concurrency-coalesced. `getSharedServerUrl(id)` is the sync read-side used at agent spawn. `shutdownSharedServers()` tree-kills every managed child (called from `gracefulShutdown` and `/api/shutdown`). Rollout is platform-gated by `isSharedHttpPlatformProven()` ‚Äî on by default on Windows, fallback to the module's stdio `mcpServer` elsewhere; `EVENT_HORIZON_SHARED_MCP=0` forces stdio everywhere. **FLUX-1004 (epic FLUX-996 "Restore Stability"):** the internal `mcpHandshakeOk(port)` fetch (now exported for direct testing) had no `AbortSignal` ‚Äî a server that accepted the TCP connection but never answered `initialize` hung it indefinitely (Node 22 undici bounds the connect phase but not the header/response wait), which could eat the whole `HANDSHAKE_TIMEOUT_MS` (45s) poll-loop budget in one attempt or hang past it. It now aborts a single attempt after `HANDSHAKE_FETCH_MS` (5s), distinct from the 45s wall-clock loop bound. Separately, `claude-code.ts`'s `ensureSharedServersForRoot` ‚Äî awaited on the spawn/chat-turn request path before every Claude spawn/resume ‚Äî no longer waits the full 45s for a cold Serena start: each module's readiness races against `PRE_SPAWN_WAIT_CAP_MS` (5s) via `raceWithCap`, and a still-starting server's own promise keeps running in `ensureSharedServer`'s `inflight`/`servers` maps regardless (so the next turn finds it ready without restarting it). A server not ready within the cap is simply omitted from that turn's MCP config ‚Äî `buildModuleServerMap` already degrades gracefully on a proven platform (no stdio fallback, just skip), so this doesn't change behavior on a timeout, only how long the caller waits before proceeding without it.

-   `engine/src/skill-installer.ts` is the CLI entrypoint used by the workspace `install-skill` command. `parseArgs` accepts `--target` / `--framework` / `--dry-run` / **`--force`** (FLUX-882), threading `force` into `installWorkspaceWorkflow`. Because the orphan sweep already runs on *every* install (idempotent + EH-scoped), `force` is **log-only** ‚Äî it records that a hard reinstall was requested and guarantees the sweep, but does not change what is written. The **auto self-heal** lives in `bootstrap.ts` `installSkillsForWorkspace`: on **every** workspace activation it calls `installWorkspaceWorkflow` (no version gate, no `checkSkillVersionStaleness` read, no one-shot marker), which overwrites the managed skill/instruction files AND runs `cleanOrphanedSkillFiles` ‚Äî so a skill module renamed or removed by an older install (e.g. the 34‚Üí24 tool consolidation) is swept away without any migration apparatus. The sweep is idempotent and strictly EH-scoped, so running it unconditionally every activation keeps the surface self-healing; the install is best-effort (a failure is logged and never blocks boot).

-   `engine/src/release.ts` is the **land-here-first** module for `npm run flux:release <version>` (CLI, `tsx src/release.ts`): it moves every `Done` ticket to `Released`, writes the per-version release notes file (`<releaseNotesPath>/<version>.md` or a single rolling file, per `releaseSettings.generateDistinctFiles`), and ‚Äî since **FLUX-1151** ‚Äî appends a block to a separate, canonical, append-only **`<releaseNotesPath>/INDEX.md`** (default `.docs/release-notes/INDEX.md`): one `- **{id}**: {title}[ ‚Äî {gist}]` line per released ticket, where `{gist}` is derived from the ticket's most recent `type:'comment'` history entry (the completion comment `finish_ticket` writes) via the exported `deriveGist` ‚Äî falling back to title-only when absent (portal-closed ticket, `merge_tickets` fold, or pre-convention history). `buildDoneIndexBlock`/`hasExistingVersionBlock` are the other two exported pure helpers (unit-tested in `release.test.ts`); `hasExistingVersionBlock` is a cheap operator-error guard ‚Äî re-running the same version logs a warning and skips appending a duplicate block instead of a read-modify-rewrite. This is the first stop for the grooming/implementation skills' FLUX-1048 "reground before starting" check ("was this already done?") ‚Äî cheaper than a `list_tickets` sweep or reading a 150-ticket per-version notes file line by line, though it only covers already-*released* work (a `Done`-but-unreleased ticket still needs the `list_tickets(status:'Done')` sweep) and was not backfilled for releases before FLUX-1151.

-   `engine/src/branch-manager.ts` owns all git plumbing for per-ticket branches: create, status, delete (local + remote), PR creation via `gh`, and the `finish_ticket` diff capture (numstat summary + unified-diff sidecar at `<flux-dir>/<ID>.diff`). `getPullRequestStatus(branch)` (FLUX-556) normalizes `gh pr view --json` into `PrStatus` (number/state/url/title/reviewDecision/mergeable + a `{total,passed,failed,pending}` check tally), returning null when no PR / gh unavailable ‚Äî the read side of the in-EH PR card. **FLUX-998 (epic FLUX-996):** every git/gh call here now routes through `git-exec.ts`'s `runGit`/`runGh` (the internal `git()`/`gitDiff()` helpers delegate to `runGit`) instead of a bare `execFileAsync` ‚Äî timeout + non-interactive env + tree-kill on every branch create/push/PR raise/merge, which used to hang the spawn/Ready/finish paths forever. **`checkGhAuth()` is the one deliberate exception**: it passes `{env: process.env}` to `runGh` instead of the default (`buildGitSyncEnv()`-built) env, because `buildGitSyncEnv()` itself calls `checkGhAuth()` to decide whether to inject gh's credential helper ‚Äî routing `checkGhAuth`'s own probe through the normal path recurses infinitely (a real `RangeError: Maximum call stack size exceeded` hit during this migration, now locked by a regression test in `branch-manager.test.ts`). Do not remove that override.

-   `engine/src/pr-tickets.ts` is the **land-here-first** module for **PR-as-a-first-class-ticket** (FLUX-566, epic FLUX-565). `syncPrTickets(workspaceRoot)` (wired into the same 90s poll as `reconcilePullRequests`) runs `gh pr list` and **upserts an engine-managed `PR-<n>` ticket** (`kind:'pr'`) per open PR via `task-store.upsertManagedTicket` (idempotent ‚Äî writes only on a real change): it carries the PR metadata (`prNumber`/`prState`/`reviewDecision`/`isDraft`/`implementationLink`) and its **work-gated `members`** (normal tickets on the PR's branch that are In Progress/Ready ‚Äî `selectMembers`, pure + tested). State mapping (decision #3) ‚Äî `syncPrTickets` is the **PR-lifecycle owner** (FLUX-569): a NEW open PR ‚Üí **Ready**; **CHANGES_REQUESTED ‚Üí In Progress + `changes-requested` tint and its Ready members are bounced back to In Progress** (the review-fail unwind); a **reopened** PR (was Done / terminal `prState`, now OPEN again on gh) climbs back out of Done ‚Üí **Ready** with `prState` reset to OPEN (fixes the FLUX-566 deferred edge); otherwise an existing open PR keeps its status (so a send-for-review move to In Progress isn't clobbered). PR tickets whose PR left the open list ‚Üí **Done** (`prState` MERGED/CLOSED). `selectMembers` / `prTicketFields` / `prTicketId` are exported pure helpers (`pr-tickets.test.ts`). The synthetic-ticket write primitive `upsertManagedTicket` lives in `task-store.ts`. **Board rendering ‚Äî the deck (FLUX-567, P2):** `portal/src/components/PrDeckCard.tsx` is the distinct PR card (violet theme, stacked-card shadow, state/review chips, changes-requested tint) ‚Äî `Column.tsx`'s `renderTask` routes `kind:'pr'` tickets to it (grouped in the **Open PRs** section), everything else to `TaskCard`. `Board.tsx` does **deck-replace**: it excludes folded member ids (union of every PR ticket's `members`) from the per-column task lists, so members live *inside* the deck (unwind to reveal them with their real status), not in their own columns. A non-PR ticket whose `branch` is in `AppContext.prByBranch` (a PR's branch) but isn't folded ‚Äî a Todo/Grooming/Backlog "pile" ticket ‚Äî gets a `‚Üí PR-n` marker in place (decision #4: never leaves its column). **Surface + actions (FLUX-568, P3):** `PrDeckSection`'s action bar is **state-aware** ‚Äî when changes were requested (`swimlane === 'changes-requested'`) **Continue development** leads (launches a dev agent in the *implementation* phase on the PR branch) and Merge drops to secondary; otherwise **Review** (implementation‚Üíreview launcher) + Merge (primary). Both route through the controller's `openLauncherInPhase(phase, templateId?)` so the launcher phase is set deterministically. Members stay folded during the fix loop (Review ‚Üí changes-requested ‚Üí Continue development ‚Üí push ‚Üí re-review ‚Üí Merge). `TaskModalPopupView` renders `PrPanel` (full diff/status/send-for-review/update-branch) for any `kind:'pr'` ticket regardless of status. **Generalized deck ‚Äî epics fold their subtasks (FLUX-580):** the fold/unwind UI is extracted into `portal/src/components/TaskDeck.tsx` (a `{ id, items, label, accent: 'violet'|'indigo' }` primitive ‚Äî the `Layers` toggle + compact `hideStatusBadge` member cards, default folded) and now backs BOTH the PR deck (`PrDeckSection` renders `<TaskDeck accent="violet">`) and the **epic ‚Üí subtasks** deck (`TaskCard`'s epic branch renders `<TaskDeck accent="indigo">` below `CardSubtaskProgress`). The shared fold rule lives in `portal/src/lib/decks.ts` (single source of truth, so Board and the card never drift): `collectPrMemberIds` (PR members), `isFoldedIntoEpic`/`epicDeckSubtasks` (a subtask folds into its epic **only when same status/column** AND not already a PR member ‚Äî **PR precedence**), and `collectEpicFoldedIds` (the board-wide union). `Board.tsx`'s deck-replace now excludes `foldedMemberIds ‚à™ epicFoldedIds` from the columns; `useTaskCardController` exposes `epicFoldedSubtasks` and `AppContext`/`appStore` carry a memoized `prMemberIds` set so the card applies PR precedence without re-scanning every PR. Cross-column subtasks stay in their own column (unchanged).

-   `portal/src/components/EpicsScreen.tsx` is the **land-here-first** component for the **epic / roadmap view** (FLUX-678) ‚Äî a top-level `AppView` (`'epics'`, registered like the other screens: `appStore.ts` union + `AppContext.tsx` `VIEW_PATHS`/`getViewFromLocation` (`/epics`) + `App.tsx` switch branch + `Header.tsx` `Target` NavItem). It lists every epic (`isEpic` = task with ‚â•1 subtask) with its count-based completion rate, sorted by % complete ascending, and drills in to the **remaining (non-done) subtasks grouped by board status** (group order derived from `config.columns`), every row clickable ‚Üí `openTaskFullView`/`openTaskModal`. The completion math is **shared, not recomputed**: `portal/src/lib/epics.ts` (`isEpic` / `getDoneStatuses(config)` ‚Üí {Done, Released, archive status} / `computeEpicRollup(epic, taskById, doneStatuses)` ‚Üí `{ resolvedSubtasks, done, total, pct }`) is the single source of truth, consumed by BOTH this screen and `useTaskCardController` (board card), so done/total can never drift between the two surfaces. The progress bar primitive is `portal/src/components/EpicProgressBar.tsx`, extracted from `task-card/CardSubtaskProgress.tsx` so both render the identical bar. Phase 1 = list + drill-in only; a dated timeline/Gantt (`targetDate`) is deferred Phase 2.

-   **Onboarding "What you can do" feature panels (FLUX-755)** are **data-driven from a committed JSON config** with a hidden, dev-only editor. `portal/src/config/onboardingFeatures.json` is the **single committed seed** (`{ version, features: { id, icon, title, desc }[] }`); `portal/src/config/onboardingFeatures.ts` is the **land-here-first** typed loader ‚Äî it statically imports the JSON (needs `resolveJsonModule`, now set in `portal/tsconfig.app.json`), declares `FeaturePanel`/`OnboardingFeaturesConfig`, and exports `validateFeatures()` (defensive coercion: drops non-objects, coerces fields to strings, auto-fills missing/duplicate ids, tolerates unknown icon names) + `FEATURE_PANELS`. `portal/src/config/featureIcons.ts` is the **single icon resolution path** ‚Äî `resolveFeatureIcon(name)` over lucide-react's `icons` record (NO `DynamicIcon` in 1.14.0) with a `Sparkles` fallback, plus the curated `FEATURE_ICON_NAMES` list backing the picker. `portal/src/components/onboarding/FeatureHighlights.tsx` is the **sole renderer** of the feature-card grid (markup copied verbatim from the wizard's old inline grid, keyed by `id`, icon via `resolveFeatureIcon`) ‚Äî reused by BOTH `OnboardingWizard.tsx` step 7 and the dev preview, so they are pixel-identical by construction. `portal/src/components/dev/OnboardingEditorScreen.tsx` is the **dev-only `/dev/onboarding` editor** (a `'dev-onboarding'` `AppView` registered like the other screens ‚Äî `appStore.ts` union + `AppContext.tsx` `VIEW_PATHS`/`getViewFromLocation` + `Header.tsx` `NAV_TINTS`/NavItem + `App.tsx` render ‚Äî but every render/route/nav site is **`import.meta.env.DEV`-gated** so the screen, its `lazy` import, and its dnd-kit usage are tree-shaken from the prod bundle and a hand-typed `/dev/onboarding` falls through to the board): LEFT = dnd-kit reorder (ChatDock pattern, vertical, id-keyed) + per-row icon picker + title/desc + add/remove + preview step/theme/width controls + Load/Save + unsaved badge; RIGHT = inline step-7 chrome around the shared `<FeatureHighlights features={draft} />` + a disabled no-op Continue (NO `OnboardingWizard` is embedded ‚Äî zero wizard side-effect surface). The editor hydrates `draft` from the engine GET (the live file, fallback `FEATURE_PANELS`) and persists via PUT; after a save Vite HMR re-imports the changed JSON so an open wizard also refreshes. **Reachability gate (FLUX-755):** `App.tsx`'s `showOnboarding` (uncompleted onboarding ‚Üí `<OnboardingWizard/>`) and the no-workspace `<WorkspaceSelector/>` early returns BOTH skip when `import.meta.env.DEV && view === 'dev-onboarding'` ‚Äî without this the editor would be unreachable for the exact dev who clears `eh-onboarding-complete` to re-test the wizard (the early return would otherwise win before the Header nav link or the view branch ever render, and a hand-typed `/dev/onboarding` would be short-circuited to the wizard). The exception is dev-gated, so it cannot widen either gate in a prod build. **Engine side:** `engine/src/routes/dev-onboarding.ts` is a **dev-only** Router (`GET`/`PUT /api/dev/onboarding-features`) over the ONE hardcoded `PANELS_FILE` (`resolveSkillSourceRoot()` + `portal/src/config/onboardingFeatures.json`, NO request interpolation) with `isPathInsideRoot` + basename guards (403), PUT body-shape validation (400, write nothing), and atomic temp-write+rename. It is gated on TWO independent layers ‚Äî `engine/src/index.ts` mounts it (WITHOUT `requireWorkspace`, since the file is repo-relative not workspace-relative) only when `DEV = !isPackaged && NODE_ENV !== 'production'`, and each handler re-checks `DEV` and 404s as a backstop. The portal API client adds `fetchOnboardingFeatures()`/`saveOnboardingFeatures()` in `portal/src/api.ts`. **These dev endpoints are intentionally NOT added to `.docs/event-horizon/reference/rest-api.md`** because they never ship to production (no packaged build mounts them).

-   **Data-driven onboarding flow (FLUX-756, Phase 1)** turns the WHOLE onboarding wizard data-driven: it renders from a committed flow config instead of a hardcoded `step===N` switch, preserving today's 9-page behavior byte-for-byte. `portal/src/config/onboardingFlow.json` is the **committed seed** (`{ version: 2, pages: OnboardingPage[] }` ‚Äî a 1:1 transcription of the old 9 steps, copy verbatim). `portal/src/config/onboardingFlow.ts` is the **land-here-first** typed loader for the flow ‚Äî it statically imports the JSON, declares the flow types (`OnboardingPage`/`OnboardingFlowConfig`/`OnboardingPageKind`/`OnboardingWidgetId`/`OnboardingCondition`/`OnboardingImage`/`OnboardingFeaturesRef`/`OnboardingCta`), and exports `SYSTEM_PAGE_SPECS` (the **single source of truth** for the 7 functional-step rails ‚Äî `system`/`required`/`locked`/`dependsOn` per widget id), `DEFAULT_FLOW` (the seed mirrored in code), `validateFlow()`, and `FLOW = validateFlow(rawFlow)`. `validateFlow` mirrors `validateFeatures`' defensive posture (drop non-objects, coerce id/title to strings, auto-fill blank/dup ids as `page-<idx>`, default `kind` to `'content'` when `widget` is absent/unknown) and adds the **functional-step backstop**: it MERGES the matching `SYSTEM_PAGE_SPECS` row onto every `kind:'widget'` page (a hand-edit can never strip `required`/`locked`/`dependsOn`/`system`), runs a **topological check** (every widget page's `dependsOn` predecessors must appear earlier ‚Äî on violation it re-derives the canonical SYSTEM ordering while keeping content pages in their authored slots), **re-injects** any missing required system page from `DEFAULT_FLOW`, and falls back to `DEFAULT_FLOW` on a missing/garbage file (incl. a legacy `{version,features}` shape with no `pages[]`). -   **Substrate vs projection (FLUX-658).** `engine/src/transcript.ts` is the immutable, append-only **RAW substrate of record** ‚Äî one JSONL file per stream at `<fluxDir>/transcripts/<streamId>.jsonl`. Legacy pre-envelope lines read losslessly: `readTurns` wraps them on the fly (`seq` = line index), discriminating an envelope from a bare event by the `turnId`+`seq`+`raw` trio. -   `engine/src/hitl-prompts.ts` is the **land-here-first** module for the **restart-durable human-in-the-loop prompt store** (FLUX-833) shared by both HITL channels ‚Äî gated-tool approvals (`permission-prompts.ts`, FLUX-605) and structured questions (`ask-questions.ts`, FLUX-662). -   `portal/src/transcriptCache.ts` is the **land-here-first** module for **chat transcript render
	latency across minimize/reopen** (FLUX-750). A module-level LRU (`Map`, insertion order = recency,
	cap 12) keyed by `conversationId` with `getTranscript` / `hasTranscript` / `setTranscript`. The
	dock minimizes a chat by *unmounting* its `<ChatWindow>`, which destroys `useChatSession`'s local
	`messages` state ‚Äî so a reopen used to cold-fetch from disk and flash blank. The hook now
	lazy-hydrates `messages` from this cache on mount (instant render, SWR-revalidated by the existing
	mount fetch) and writes through on every successful fetch; a true cache miss flips a `loading` flag
	that drives the cold-open spinner in `ChatView`. `ChatDock` keeps the cache fresh for a *running
	but minimized* conversation via a bounded warm subscription (only `running` ids not in `open`, no
	DOM, no per-idle-chat cost) so reopening a live session shows current committed state with no
	stale-then-jump. Pure module scope (no React) ‚Üí survives unmount, resets on full reload ‚Äî the same
	lifetime as `DockProvider`'s in-memory `drafts` map. `liveText` mid-stream tokens are intentionally
	NOT cached (the durable transcript via the warm-sub satisfies "current committed state immediately").

-   `portal/src/chatQueueCache.ts` is the sibling **land-here-first** module for the **FLUX-748 mid-turn
	message queue across minimize/reopen** (FLUX-837). Same lifetime trick as `transcriptCache` (module
	`Map` keyed by `conversationId`, survives the `<ChatWindow>` unmount, resets on full reload) but for
	the queue of messages the user submitted while a turn was `working`/`busy` ‚Äî those live in
	`useChatSession`'s local `queued` state, so minimizing mid-turn used to *destroy an un-sent message*
	(it was queued, not yet in the durable transcript). The hook now lazy-hydrates `queued` from this
	cache, writes through on change, re-seeds on a conversationId switch, and ‚Äî on a fresh reopen with a
	parked queue and a confirmed-idle session ‚Äî dispatches the head once (the edge auto-dispatch can't
	fire on mount, so a turn that finished while minimized would otherwise leave the message parked).

## Repo-backed data surfaces

-   `.flux/*.md` are the canonical ticket files.
    
-   `.flux/config.json` defines statuses, hidden statuses, tags, priorities, and workflow status names.
    
-   `.docs/**/*.md` are the project docs exposed inside the portal.
    
-   `.flux/skills/event-horizon-agent.md` is the source skill document.
    
-   `.flux/skills/event-horizon-copilot-instructions.md` is the source always-on Copilot instructions template.

-   `.flux-store/*.md` are the ticket files in **orphan-branch storage mode** ‚Äî a git worktree on the `flux-data` branch that the engine auto-syncs (commit ‚Üí fetch ‚Üí merge ‚Üí push) via `engine/src/sync-watcher.ts`. `runSync` is the **land-here-first** code for sync/merge behavior: it must **never `git add -A`/commit while a merge is unresolved** (MERGE_HEAD set or unmerged paths present) ‚Äî doing so bakes git conflict markers into a ticket's YAML and corrupts it (FLUX-703/FLUX-694). On a conflicting merge it parks in `state:'conflict'` for user resolution (`resolveConflicts`); an in-memory `pendingConflicts` short-circuit plus an on-disk `hasUnmergedState` guard keep that safe across watcher self-triggers and engine restarts. A file that still ends up with committed `<<<<<<<`/`=======`/`>>>>>>>` markers is reported by `task-store.ts` `loadTask` as "unresolved git conflict markers" rather than a cryptic YAML error.
