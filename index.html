        Push notifications

        Lumen fans transaction events into per-user push digests. Clients register one device token per install.

        Card transactions now batch into a 30s digest window — one push per burst of spending, not one per swipe. Consumers should prefer digest webhooks over per-event pushes.

        no lock-in

        A ticket is just a file.

        Markdown with YAML frontmatter, committed to your repo or a dedicated data branch. Which means your board:

        • versions with your code — history is git history
        • works offline — local-first, no service dependency
        • moves with you — switch machines with a git clone
        • stays greppable — search it like any other source
        .flux/tickets/LUM-229.md
        ---
id: LUM-229

branch: lum-229-push-digests

tags: [mobile, notifications]

        storage & sync

        Your board is commits. You pick where they live.

        Tickets are markdown — so sync is just git. Keep them next to your code, on a dedicated orphan branch that never touches master's history, or in a local store with no git at all.

        where tickets live — three modes, same markdown
        lumen-app/.flux/tickets/LUM-242.md
        • Board changes ride your normal commits — tickets version with the code they describe.
        • A feature PR can include its own ticket's status change.
        • Best for solo repos and teams that like everything in one history.
        your-repo · branch flux-data (orphan) · .flux-store/tickets/LUM-242.md
        • An orphan branch in the same repo — no shared history with master, so your code log stays pristine.
        • Board syncs with plain push / pull — same remote, same auth, no extra service.
        • Every machine that can clone the repo gets the whole board. The default for teams.
        ~/.flux-store/lumen-app/tickets/LUM-242.md
        • No git at all — tickets live in a local store beside your workspaces.
        • Right for repos you can't commit board files into, or quick experiments.
        • Switch modes later — it's the same markdown either way.

        same files, three homes — the orphan branch keeps the board in the repo without touching master's history
